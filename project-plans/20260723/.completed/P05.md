## Lifecycle Order Evidence

Verified the exact ordered lifecycle sequence:

```
setup → C init → game → run_end attempt → automation output closed
→ teardown_subsystems returned → teardown_complete → status mapping
```

Evidence from `rust/src/main.rs` and `rust/src/automation/lifecycle.rs`:

1. **Setup** (`main.rs:69`): `setup_automation(&auto_opts, &caps)` is called
   before `run_uqm`. Validates script, creates output root, checks build caps.
2. **C init** (`lifecycle.rs:120`): `lifecycle.c_init()` — init result checked;
   failure (positive) returns early before game loop; negative (version/usage)
   returns 0.
3. **Game** (`lifecycle.rs:138`): `lifecycle.run_game()` — game result captured.
4. **run_end attempt** (`lifecycle.rs:142`): `rt.finalize()` — atomically changes
   phase to Finalizing, writes run_end exactly once (duplicate rejected).
5. **Automation output closed** (`lifecycle.rs:142-143`): `rt.finalize()` clears
   capture generation, drains active shells/reservations, closes handles. Late
   callback cannot use writer (`can_write() == false` after finalize).
6. **teardown_subsystems returned** (`lifecycle.rs:149`): Called with no
   automation mutex/I/O lock held (finalize already dropped all handles).
7. **teardown_complete** (`lifecycle.rs:152-156`): Receipt written only after
   `teardown_subsystems()` returns. Uses durable file helper
   (create_new → write → flush → sync → close → exclusive publish).
8. **Status mapping** (`lifecycle.rs:159`): `map_status(terminal, game_result)`
   — zero only for Success/CooperativeStop with zero game result.

## Requirement Verification

### REQ-MODE-001 — Runtime opt-in
**PASS.** `main.rs` calls `setup_automation()` before `run_uqm()` (line69 before
line83). When both options supplied, script is parsed/validated and output root
created before any game initialization.

### REQ-MODE-002 — Complete pair
**PASS.** `setup.rs::setup_automation()` returns `Err` for `is_incomplete()`.
`main.rs` calls `exit(1)` on error. Verified: incomplete pair fails before
game initialization and artifact creation. Two test variants (both directions).

### REQ-MODE-003 — Inactive behavior
**PASS.** `runtime.rs::shell_enter()` performs only:
- Saturating lock-free ABI-entry increment (`fetch_update` with `checked_add`)
- Acquire activation load via `mirror.is_active()` (lock-free AtomicU8)
- Returns `inactive_no_stop()` immediately

No TLS/depth access, no allocation, no formatting/logging, no lock (verified
by deadlock test that holds mutex then calls shell_enter), no C/SDL/I/O call,
no input/activity mutation, no active gate entry, no scheduling, no artifacts.
The inactive fast path uses `mirror.is_active()` (lock-free) not `inner.lock()`.

### REQ-BUILD-001 — Supported linked configuration
**PASS.** `setup.rs::REQUIRED_BUILD_FLAGS` lists exactly: RUST_OWNS_MAIN,
USE_RUST_THREADS, USE_RUST_GFX, USE_RUST_COMM, USE_RUST_RESTART (NOT
USE_RUST_RESOURCE). `is_supported()` checks all five. Unsupported configurations
return `Err` before game init.

### REQ-BUILD-002 — No fictional flag
**PASS.** No `RUST_AUTOMATION` Cargo feature in `rust/Cargo.toml`. Confirmed by
search: `[features]` section contains only `default`, `audio_heart`,
`debug-process`.

### REQ-EXIT-006 — Outer Rust guard
**PASS.** `check_terminal_guard()` checks sticky terminal at outer boundaries.
`reassert_abort_if_terminal()` reasserts abort when terminal. Both tested with
terminal and non-terminal states.

### REQ-EXIT-008 — Existing cleanup then status
**PASS.** `run_lifecycle()` unwinds through game cleanup and
`teardown_subsystems()` before status mapping. `map_status()` returns zero only
for fully evidenced success (Success/CooperativeStop with zero game result).
All failure classes map to nonzero.

### REQ-EXIT-009 — Post-teardown evidence
**PASS.** `run_end` is finalized (`rt.finalize()`) and all automation handles
are closed before teardown. Receipt (`teardown-complete.json`) is written only
after `teardown_subsystems()` returns. The `teardown_happens_before_receipt`
test proves no marker handle is open across teardown by checking receipt
existence during teardown callback.

### REQ-FFI-005 — Two-phase commit and finalization race
**PASS.** Finalization:
- Clears activation/capture (`clear_capture_generation()`)
- Drains active shells/reservations (`ShellsStillActive` result blocks finalize)
- Takes ownership exactly once (`Finalized`/`AlreadyFinalized`/`AlreadyFinalizing`)
- Denies late access to dropped resources (`can_write() == false`, `reserve_transition() == None`)
- Duplicate run_end rejected (`DuplicateRunEnd`)

### Ordered lifecycle evidence (item 10)
**PASS.** Full sequence verified as documented above.
