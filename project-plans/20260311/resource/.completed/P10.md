### C-Side Analysis

#### `sc2/src/libs/resource/rust_resource.h` and `rust_resource.c`
**Status**: Present in build system but conditionally compiled

**Evidence:**
1. **Makeinfo listing**: Both files are in `uqm_CFILES` and `uqm_HFILES` unconditionally
   ```
   uqm_CFILES="... rust_resource.c"
   uqm_HFILES="... rust_resource.h"
   ```

2. **Conditional compilation**: Both files wrapped in `#ifdef USE_RUST_RESOURCE`
   ```c
   #ifdef USE_RUST_RESOURCE
   // ... all content ...
   #endif /* USE_RUST_RESOURCE */
   ```

3. **FFI function dependencies**: These C wrappers call functions from the removed `ffi.rs`:
   - `rust_init_resource_system()`
   - `rust_uninit_resource_system()`
   - `rust_load_index()`
   - `rust_get_string_resource()`
   - `rust_resource_loader_init()`
   - `rust_resource_load()`
   - `rust_resource_free()`
   - `rust_resource_exists()`
   - `rust_cache_init()`
   - `rust_cache_get()`
   - `rust_cache_insert()`
   - `rust_cache_clear()`
   - `rust_cache_size()`
   - `rust_cache_len()`

**Impact Analysis:**
- These C files **will fail to link** if `USE_RUST_RESOURCE` is defined, because `ffi.rs` has been removed
- Since the authoritative path is `ffi_bridge.rs` (not `ffi.rs`), these wrappers are targeting the wrong API
- **Recommendation**: Leave in place for now (harmless when `USE_RUST_RESOURCE` is not defined), but mark for Phase 11 cleanup or document as deprecated

**Deferred Action:**
- C-side cleanup deferred to Phase 11 (C integration verification)
