# Rust UIO Subsystem Requirements

## Scope

These requirements define the expected externally observable behavior and integration contract of the Rust UIO subsystem for UQM. They focus on externally observable behavior, except where explicit compatibility constraints, FFI safety, or ABI layout rules require more specific provisions.

The subsystem is a virtual filesystem service that presents a unified namespace across mounted directories and mounted archives, and that provides directory, file, stream, metadata, and bridge services through the established UIO API surface.

### Requirement classification

Requirements are classified by source and priority:

- **[engine-critical]** — Required for the engine to load content and run correctly. Failure blocks engine viability. These must be addressed first. Each item tagged engine-critical has explicit consumer evidence establishing the dependency.
- **[compatibility-complete]** — Required for full API parity with the legacy C implementation. Failure may not block basic startup/runtime operation, but it still breaks supported API surface for edge cases, secondary consumers, tooling, or other documented in-tree use. It must not be read as meaning “optional.” Compatibility-complete items remain mandatory for final subsystem completion even when they are not current engine-viability gates. Clean stub failure is acceptable only as an interim implementation state unless a specific requirement explicitly defines failure-only behavior as the intended long-term contract.
- **[quality/cleanup]** — Desired for code quality, safety, or maintainability. Not a compatibility blocker.

Requirements whose text explicitly says **provisional**, **interim contract**, or **acceptance rule pending legacy verification** are still normative for the current documentation baseline, but they must be interpreted as temporary acceptance scaffolding rather than as legacy-final compatibility claims. If later verification proves a different legacy behavior, the verified behavior supersedes the provisional rule. In particular, where these requirements define deterministic behavior for unresolved overlap/order/backing-type cases, that deterministic behavior is the required acceptance contract for now, not a claim that the same rule is already proven legacy parity.

Where a requirement is not explicitly tagged, it is **[engine-critical]** or **[compatibility-complete]** based on context.

### Terminology

See specification §1.1.1 for the glossary distinguishing *parity*, *compatibility-complete*, and *port-completeness*.

## Repository and initialization requirements

- **REQ-UIO-INIT-001** [engine-critical] When the UIO subsystem is initialized, it shall become ready to create repositories, open directories, mount filesystems, enumerate directories, and perform file and stream I/O. *Observed startup dependency: `uqm/setup.c` calls `uio_init()` before the setup path begins performing UIO operations. This requirement is engine-critical because the subsystem must be initialized before repository and mount work can proceed, not because `options.c` itself directly provides the observed call site.*
- **REQ-UIO-INIT-002** [quality/cleanup] When the UIO subsystem is initialized more than once, it should behave idempotently. *Rationale: Proposed design — the legacy C contract does not clearly establish that consumers depend on idempotent initialization. This is a defensive robustness goal, not a proven compatibility requirement.*
- **REQ-UIO-INIT-003** [engine-critical] When the UIO subsystem is uninitialized, it shall release subsystem-global resources that are owned by the subsystem. *Consumer dependency: engine shutdown path calls `uio_unInit()`.*
- **REQ-UIO-REPO-001** [engine-critical] When a caller opens a repository, the subsystem shall create a repository object that owns a mount namespace. *Consumer dependency: `options.c` creates exactly one repository at startup. Note: The public API permits multiple repositories, but no current consumer exercises independent multi-repository behavior. Multi-repository support is retained for API completeness.*
- **REQ-UIO-REPO-002** [engine-critical] When a caller closes a repository, the subsystem shall unmount all mounts owned by that repository and release repository-owned resources.
- **REQ-UIO-REPO-003** [engine-critical] While a repository remains open, the subsystem shall preserve mount ordering and path-resolution behavior for that repository.

## Mount and namespace requirements

- **REQ-UIO-MOUNT-001** [engine-critical] When a caller mounts a source into a repository at a virtual mount point, the subsystem shall make the mounted content addressable through the repository namespace at that mount point. *Consumer dependency: `options.c` mounts content, config, addon, and archive directories.*
- **REQ-UIO-MOUNT-002** [engine-critical] When a mount is created with top, bottom, above, or below placement semantics, the subsystem shall place the mount according to those placement semantics for the basic cases (TOP before all, BOTTOM after all, ABOVE/BELOW relative to an anchor). *Consumer dependency: `options.c` uses TOP, BOTTOM, and ABOVE placement. Note (provisional — acceptance criteria pending verification): Edge-case interactions between mount-point path specificity and insertion-based precedence are not yet fully characterized from the legacy C implementation — see specification §15, question 8. Until verified otherwise, the interim rule for those unresolved overlap cases is: explicit placement precedence first, then longest matching mount-point prefix, then recency/insertion order.*
- **REQ-UIO-MOUNT-003** [engine-critical] When multiple mounts can satisfy the same virtual path for a read operation in the non-ambiguous cases exercised by current startup and content-loading flows, the subsystem shall resolve the path using mount precedence and shall select the highest-precedence matching mount. *Consumer dependency: archive mounts override base content. Note (provisional): Edge cases where mount-point specificity competes with insertion order remain pending legacy verification — see specification §15, question 8. Until verified otherwise, the interim implementation rule for those unresolved overlap cases is: explicit placement precedence first, then longest matching mount-point prefix, then recency/insertion order. The current normative guarantee for proven parity stops at the exercised/common cases, but implementations shall still apply that interim rule consistently for deterministic behavior outside them.*
- **REQ-UIO-MOUNT-004** [engine-critical] When a mount is created as read-only, the subsystem shall reject write, create, delete, rename, and directory-modifying operations that target content provided by that mount.
- **REQ-UIO-MOUNT-005** [engine-critical] When a mount is unmounted, the subsystem shall remove that mount from subsequent path resolution and directory enumeration results.
- **REQ-UIO-MOUNT-006** [compatibility-complete] When all mounts in a repository are unmounted, the subsystem shall leave the repository with no active namespace contributions.
- **REQ-UIO-MOUNT-007** [compatibility-complete] When a caller requests the filesystem type of a mount, the subsystem shall report the type associated with that mount.
- **REQ-UIO-MOUNT-008** [compatibility-complete] When a caller transplants an existing directory handle into another mount point, the subsystem shall create a new mount exposing the same underlying content at the requested virtual location. *Consumer dependency: `options.c:582` uses `uio_transplantDir` for addon shadow-content mounting above `contentMountHandle`. No engine-critical startup dependency on transplant has been established — `prepareMeleeDir()` uses `uio_openDirRelative`, not `uio_transplantDir`. If transplant fails, addon shadow-content mounting degrades but the engine remains viable for base content loading and core gameplay.*
- **REQ-UIO-MOUNT-009** [engine-critical] While a mount is active, operations against that mount — including path resolution, directory enumeration, file opening, and file-location queries — shall continue to resolve and return results consistent with the mount's content until the mount is unmounted.
- **REQ-UIO-MOUNT-010** [compatibility-complete] When a mount request is invalid for the requested placement semantics, repository, or parameters, the subsystem shall fail the mount request without partially registering the mount.

## Path handling requirements

- **REQ-UIO-PATH-001** [engine-critical] When the subsystem resolves a virtual path, it shall normalize the path consistently so that equivalent paths address the same virtual location.
- **REQ-UIO-PATH-002** [engine-critical] When the subsystem resolves a relative path against a directory handle, it shall interpret the path relative to that handle.
- **REQ-UIO-PATH-003** [engine-critical] When the subsystem resolves an absolute virtual path, it shall interpret the path from the repository namespace root rather than from the caller's current directory handle.
- **REQ-UIO-PATH-004** [engine-critical] When a path cannot be resolved through the repository namespace for the requested operation, the subsystem shall report failure rather than silently falling back to unrelated content.
- **REQ-UIO-PATH-005** [compatibility-complete] When a caller asks for the physical location of a virtual file, the subsystem shall report both the resolved physical location and the owning mount for backing types whose current documented contract defines location success, including archive-backed content to the extent required by REQ-UIO-ARCHIVE-010. *Note: Archive-backed location success is required strongly enough to support the observed legacy helper stack, but the exact returned path/string shape for non-STDIO-backed content remains an open compatibility question.*
- **REQ-UIO-PATH-006** [compatibility-complete] When a file-location query fails, the subsystem shall not return stale or partial location outputs.

## Directory handle and lifecycle requirements

- **REQ-UIO-DIR-001** [engine-critical] When a caller opens a directory handle from a repository and path, the subsystem shall return a handle representing that virtual directory location.
- **REQ-UIO-DIR-002** [engine-critical] When a caller opens a directory handle relative to another directory handle, the subsystem shall return a handle representing the resolved relative location.
- **REQ-UIO-DIR-003** [engine-critical] While a directory handle remains live, the subsystem shall preserve the handle's repository association and resolution context.
- **REQ-UIO-DIR-004** [engine-critical] When a caller closes a directory handle, the subsystem shall release that handle and shall free the handle once the final ownership reference is released.
- **REQ-UIO-DIR-005** [compatibility-complete] When a directory handle close operation is applied to a null handle, the subsystem should behave safely and should not crash.
- **REQ-UIO-DIR-006** [compatibility-complete] When diagnostic printing of a directory handle is requested, the subsystem shall emit a representation sufficient for UIO debugging and memory-debug integration.
- **REQ-UIO-DIR-007** [compatibility-complete] When a directory handle's underlying mount is unmounted while the handle remains live, the subsystem shall satisfy the authoritative post-unmount directory-handle safety obligation defined by REQ-UIO-LIFE-003. *Note: This requirement is a directory-handle-scoped cross-reference, not a second independent acceptance obligation.*

## Directory enumeration requirements

- **REQ-UIO-LIST-001** [engine-critical] When a caller requests a directory listing, the subsystem shall enumerate entries visible at the requested virtual directory location. *Consumer dependency: `options.c` enumerates directories for package and resource-index discovery.*
- **REQ-UIO-LIST-002** [engine-critical] When the requested virtual directory is contributed by multiple active mounts that include at least one archive mount, and merged visibility is required by currently proven startup/resource consumers other than the still-unverified broader combined-source `.rmp` case, the subsystem shall merge visible entries across the contributing mounts needed for those proven cases. *Consumer dependency: archive-backed assets must become visible through ordinary listings in the startup/resource-consumer paths already established by code evidence. Current engine-critical cross-mount merged-listing scope is limited to these cases: (1) startup package/archive discovery paths, (2) startup-visible `.rmp` discovery under REQ-UIO-LIST-016/017, and (3) archive-backed asset visibility in ordinary listings where a currently proven consumer depends on that visibility. For overlapping STDIO-only mounts, merged listing remains a compatibility-complete requirement unless future evidence promotes specific cases. Broader generalized merge behavior across arbitrary mount combinations, including the unverified combined-source `.rmp` discovery case, is compatibility-complete rather than proven engine-critical parity (see REQ-UIO-LIST-015).*

- **REQ-UIO-LIST-003** [engine-critical] When multiple contributing mounts provide entries with the same name, the subsystem shall return a deduplicated listing by entry name.
- **REQ-UIO-LIST-004** [compatibility-complete] When a caller supplies an empty match pattern, the subsystem shall treat the pattern as matching all entries.
- **REQ-UIO-LIST-005** [compatibility-complete] When a caller requests literal matching, the subsystem shall include only entries whose names exactly match the pattern.
- **REQ-UIO-LIST-006** [compatibility-complete] When a caller requests prefix matching, the subsystem shall include only entries whose names begin with the pattern.
- **REQ-UIO-LIST-007** [compatibility-complete] When a caller requests suffix matching, the subsystem shall include only entries whose names end with the pattern.
- **REQ-UIO-LIST-008** [compatibility-complete] When a caller requests substring matching, the subsystem shall include only entries whose names contain the pattern.
- **REQ-UIO-LIST-009** [compatibility-complete] When a caller requests regex matching, the subsystem shall provide general regex semantics compatible with the established UIO match contract rather than only the currently proven engine-critical hard-coded cases. *Note: Full POSIX-ERE-style coverage is compatibility-complete API parity, not a prerequisite for current engine viability beyond the regex cases already proven by startup consumers.*
- **REQ-UIO-LIST-010** [compatibility-complete] When no entries match a valid listing request, the subsystem shall return an empty listing object rather than reporting failure solely because the result set is empty.
- **REQ-UIO-LIST-011** [engine-critical] When a listing request cannot be performed because the target directory cannot be resolved or accessed, the subsystem shall report failure.
- **REQ-UIO-LIST-012** [engine-critical] When a listing object is freed, the subsystem shall release all memory owned by that listing, including entry-name storage and any backing buffers.
- **REQ-UIO-LIST-013** [engine-critical] While a listing object remains live, all name pointers exposed through that object shall remain valid.
- **REQ-UIO-LIST-015** [compatibility-complete] When the broader combined-source `.rmp` case is exercised — specifically, `.rmp` resource index files become visible from both STDIO and archive contributors within the same merged namespace after archive mounting — the subsystem shall support deterministic merged `.rmp` discovery with correct deduplication by name under the documented provisional listing rule. *Consumer dependency: `loadIndices()` in `options.c` enumerates `.rmp` files to discover resource indices after archive mounting completes, but whether production content actually relies on `.rmp` files existing inside ZIP/UQM archives (as opposed to only in STDIO-backed directories) is not confirmed by static inspection — see specification §15, question 9. Combined STDIO+archive `.rmp` discovery therefore remains an open compatibility question for parity purposes; supporting that combined case is a design-forward target, not a proven current-production dependency. Interim acceptance rule (provisional): until legacy merged-list ordering is verified, tests shall require deterministic `.rmp` enumeration under the chosen provisional rule of mount-precedence ordering with first-seen deduplication and lexical-by-entry-name ordering within each contributing mount. This deterministic rule is current reproducibility scaffolding for implementation and testing, not yet a claim that lexical intra-mount ordering is final legacy parity. If future legacy verification shows that this combined-source case is in fact part of production parity, its priority must be elevated accordingly. The order in which `.rmp` files appear in the listing determines `LoadResourceIndex()` processing order, so any future legacy verification that contradicts the provisional rule must override it.*

- **REQ-UIO-LIST-016** [engine-critical] When `.rmp` resource index files are enumerated from the startup-visible namespace after archive mounting, the subsystem shall preserve the proven startup behavior of discovering the `.rmp` files that are visible through the currently established listing contract, with correct deduplication by name. *Consumer dependency: `loadIndices()` in `options.c` enumerates `.rmp` files to discover resource indices after archive mounting completes. This requirement is intentionally narrower than REQ-UIO-LIST-015: it covers proven startup `.rmp` visibility and deduplication behavior, not the still-unverified broader combined STDIO+archive merged-source case and not any legacy-final ordering rule that has not yet been verified.*
- **REQ-UIO-LIST-017** [engine-critical] When `.rmp` resource index files are enumerated from the startup-visible namespace after archive mounting and the legacy-final merged-list ordering remains unverified, the subsystem shall apply the documented provisional deterministic ordering rule for acceptance and testing purposes until legacy verification confirms or replaces it. *Note: This requirement defines the current provisional ordering contract; it does not claim that the provisional rule is already proven legacy parity. If future verification establishes a different legacy ordering, that verified behavior supersedes the provisional rule.*

### Current merged-list acceptance matrix

| Scenario | Backing types involved | Current status | Ordering / dedup rule | Basis |
|---|---|---|---|---|
| Package/archive discovery in startup paths | STDIO directories containing archive files | Required now | Visible entries by mount precedence with first-seen dedup; no additional legacy-final intra-mount rule established here | Proven startup consumer |
| Startup-visible `.rmp` discovery in currently exercised namespace | Startup-visible contributors after archive mounting | Required now | Dedup by name; provisional deterministic rule uses mount precedence + first-seen dedup + lexical-by-entry-name within each contributing mount | REQ-UIO-LIST-016/017 |
| Archive-internal ordinary per-directory enumeration within a mounted archive namespace | Archive namespace only | Required now once archive support is functional | Ordinary directory enumeration with applicable precedence/dedup rules | REQ-UIO-ARCHIVE-002 |
| Ordinary listings where a proven current consumer depends on archive-backed assets being visible through merged namespace | Archive + other active contributors | Required now | Dedup by name; higher-precedence visible entry wins wholesale, including entry kind | REQ-UIO-LIST-002/003 |
| Overlapping STDIO-only mounts with no proven current consumer dependency | STDIO + STDIO | Deferred / compatibility-complete | Deterministic behavior required, but legacy-final merged-list parity not yet elevated to engine-critical | Compatibility-complete scope |
| Broader combined-source `.rmp` discovery from both STDIO and archive contributors beyond proven startup-visible case | STDIO + archive | Deferred / compatibility-complete | Provisional deterministic `.rmp` rule in REQ-UIO-LIST-015 until legacy verification confirms or replaces it | Interim acceptance rule |



## File open, read, write, and seek requirements

- **REQ-UIO-FILE-001** [engine-critical] When a caller opens a file relative to a directory handle, the subsystem shall resolve the file through the virtual namespace and return a file handle if the file can be opened with the requested access.
- **REQ-UIO-FILE-002** [engine-critical] When a caller requests read access to a readable file, the subsystem shall permit subsequent read operations through the returned handle.
- **REQ-UIO-FILE-003** [engine-critical] When a caller requests write, create, append, truncate, or read-write access to writable content, the subsystem shall apply those semantics in accordance with the public open flags and stream modes.
- **REQ-UIO-FILE-004** [engine-critical] When a caller requests write-capable access to content that is provided by a read-only mount or archive-backed content that is not writable, the subsystem shall fail the open request.
- **REQ-UIO-FILE-005** [engine-critical] When a caller closes a file handle, the subsystem shall release resources owned by that handle.
- **REQ-UIO-FILE-006** [engine-critical] When a caller reads from a file handle, the subsystem shall return the number of bytes actually read, including short reads at end of file.
- **REQ-UIO-FILE-007** [engine-critical] When a caller writes to a writable file handle, the subsystem shall return the number of bytes actually written.
- **REQ-UIO-FILE-008** [engine-critical] When a caller reads at end of file, the subsystem shall report end-of-file behavior consistent with the public API contract.
- **REQ-UIO-FILE-009** [engine-critical] When a caller seeks within a seekable file handle, the subsystem shall update the file position according to the requested origin and offset.
- **REQ-UIO-FILE-010** [engine-critical] When a caller requests file metadata by file handle, the subsystem shall report file metadata sufficient for consumers that depend on size and file-type information.
- **REQ-UIO-FILE-011** [engine-critical] When a caller requests file metadata by virtual path, the subsystem shall resolve the path through the mount namespace and report metadata for the resolved object.
- **REQ-UIO-FILE-012** [compatibility-complete] When a caller requests access checks, the subsystem shall evaluate the requested access mode semantics rather than only existence.
- **REQ-UIO-FILE-013** [compatibility-complete] When a caller renames or moves a filesystem object, the subsystem shall perform the rename only if both source and destination satisfy the operation's mount and access constraints.
- **REQ-UIO-FILE-014** [engine-critical] When a caller unlinks a file, the subsystem shall delete the target only if the resolved target is deletable under the mount's access rules.
- **REQ-UIO-FILE-015** [compatibility-complete] When a caller creates a directory, the subsystem shall create the directory only if the resolved target location permits directory creation.
- **REQ-UIO-FILE-016** [compatibility-complete] When a caller removes a directory, the subsystem shall remove the directory only if the resolved target location permits directory removal.

## Stream API requirements

- **REQ-UIO-STREAM-001** [engine-critical] When a caller opens a stream, the subsystem shall return a buffered stream object that satisfies the established `uio_Stream` contract used by C and Rust consumers.
- **REQ-UIO-STREAM-002** [engine-critical] While a stream remains open, the subsystem shall preserve buffer invariants required by the public `uio_Stream` layout and semantics.
- **REQ-UIO-STREAM-003** [engine-critical] When a caller closes a stream, the subsystem shall flush required buffered writes, close the underlying file handle, and release stream-owned resources.
- **REQ-UIO-STREAM-004** [engine-critical] When a caller performs a block read through the stream API, the subsystem shall return the number of complete items read in accordance with the `size` and `count` arguments.
- **REQ-UIO-STREAM-005** [engine-critical] When a stream read encounters end of file before satisfying the full request, the subsystem shall set the stream EOF indicator.
- **REQ-UIO-STREAM-006** [engine-critical] When a stream operation encounters an I/O error, the subsystem shall set the stream error indicator.
- **REQ-UIO-STREAM-007** [engine-critical] When a caller queries the EOF indicator, the subsystem shall report the stream's actual EOF state rather than a constant or synthesized value unrelated to stream state. *Consumer dependency: `sdluio.c` uses `uio_ferror` after zero-byte reads to distinguish EOF from error; correct EOF reporting is the other half of that distinction.*
- **REQ-UIO-STREAM-008** [engine-critical] When a caller queries the error indicator, the subsystem shall report the stream's actual error state rather than a constant or synthesized value unrelated to stream state. *Consumer dependency: `sdluio.c` calls `uio_ferror(stream)` after zero-byte reads to set `SDL_SetError` with `strerror(errno)`.*
- **REQ-UIO-STREAM-009** [compatibility-complete] When a caller clears stream status, the subsystem shall clear both EOF and error indicators. *Rationale: `uio_clearerr` is part of the stream-status contract and is needed for correct status recovery. However, no current engine-critical caller dependency on retry-after-clear has been demonstrated in the inspected code. Downgraded from engine-critical to compatibility-complete pending evidence of a caller that depends on clearing status before retrying an operation.*
- **REQ-UIO-STREAM-010** [engine-critical] When a caller seeks on a stream, the subsystem shall update the visible stream position and clear EOF status on successful repositioning.
- **REQ-UIO-STREAM-011** [engine-critical] When a caller queries stream position, the subsystem shall return the caller-visible stream position, including any effects of stream buffering.
- **REQ-UIO-STREAM-012** [engine-critical] When a caller reads a line through the stream API, the subsystem shall read through newline or buffer limit, shall null-terminate the result on success, and shall report no-data EOF distinctly from a successful short line read.
- **REQ-UIO-STREAM-013** [engine-critical] When a caller reads one character through the stream API, the subsystem shall return the next byte value on success and EOF on end-of-file or error, with status indicators updated accordingly.
- **REQ-UIO-STREAM-014** [compatibility-complete] When a caller pushes back a character, the subsystem shall provide at least the public minimum pushback behavior expected by consumers. *Note: UQM does not currently exercise `ungetc` on non-seekable streams. The pushback-slot implementation is a correctness requirement for the `ungetc` contract but not an urgent engine blocker.*
- **REQ-UIO-STREAM-015** [engine-critical] When a caller writes one character or a string through the stream API, the subsystem shall write the requested data or report failure.
- **REQ-UIO-STREAM-016** [engine-critical] When a caller flushes a stream, the subsystem shall commit buffered writes required by the stream contract.
- **REQ-UIO-STREAM-017** [compatibility-complete] When a caller requests the underlying handle of a stream, the subsystem shall return the associated handle without transferring ownership.
- **REQ-UIO-STREAM-018** [quality/cleanup] When the engine links against UIO in Rust mode, the subsystem should export the expected `uio_*` stream symbols directly from Rust so that C forwarding shims are not required. *Rationale: The current C shim for `uio_fread` is a transitional artifact. Its presence does not cause a caller-visible behavioral defect — the observable contract is the same regardless of whether the symbol is provided by Rust or via the shim. Eliminating the shim is a port-completeness goal (see specification §1.1.1), not a functional parity requirement.*
- **REQ-UIO-STREAM-019** [compatibility-complete] When formatted output through the stream API is requested, the subsystem shall either implement the required behavior directly or provide an integration-compatible path that preserves the documented external behavior. *Note: `uio_vfprintf` is used for diagnostics/debug only in UQM. No proven production dependency. This is classified as compatibility-complete, not engine-critical.*

## Archive and package requirements

### Minimum end-to-end acceptance criterion

All archive requirements below roll up to one testable acceptance criterion:

> **REQ-UIO-ARCHIVE-ACCEPT** [engine-critical] After C startup code mounts package archives via `uio_mountDir` with `uio_FSTYPE_ZIP`, engine consumers — including `sdluio.c`, Rust sound decoders, and the Rust resource bridge — shall be able to discover, open, read, seek, and query archive-backed assets through the standard exported UIO APIs without requiring any special-case behavior or caller-side awareness that the content is archive-backed.

Verification: mount a real `.uqm` package archive, enumerate its contents via `uio_getDirList`, open a known asset via `uio_fopen`, read its full contents via `uio_fread`, verify seek/tell consistency, and confirm that `uio_feof`/`uio_ferror` report correct status.

### Detailed archive requirements

- **REQ-UIO-ARCHIVE-001** [engine-critical] When a caller mounts archive content as a ZIP/UQM filesystem type, the subsystem shall register the mount as fully active and shall expose archive entries within the virtual namespace at the requested mount point. *Consumer dependency: `options.c` mounts `.zip`/`.uqm` packages via `uio_mountDir` with `uio_FSTYPE_ZIP`.*
- **REQ-UIO-ARCHIVE-002** [engine-critical] While an archive mount is active, the subsystem shall include archive-backed content in path resolution and in ordinary per-directory enumeration within the mounted archive namespace, plus the acceptance-bounded merged-directory cases actually required by current startup and resource consumers, according to the applicable mount precedence rules. Broader generalized merged-directory semantics across arbitrary overlapping mount combinations remain compatibility-complete.
- **REQ-UIO-ARCHIVE-003** [engine-critical] When a caller opens a file that resolves to archive-backed content, the subsystem shall provide read access to the decompressed file contents through the normal file and stream APIs. The resulting handle or stream shall support read, seek, tell, fstat, EOF, and error semantics equivalent to handles over STDIO-backed content. *Consumer dependency: `sdluio.c`, Rust sound decoders, and the Rust resource bridge all read content that may be archive-backed.*
- **REQ-UIO-ARCHIVE-004** [engine-critical] When a caller requests metadata for archive-backed content, the subsystem shall report metadata compatible with consumer expectations, including uncompressed file size and appropriate file-type indicators.
- **REQ-UIO-ARCHIVE-005** [engine-critical] When a caller attempts to modify archive-backed content through write-oriented APIs, the subsystem shall fail the request.
- **REQ-UIO-ARCHIVE-006** [engine-critical] When archive files are discovered by existing engine startup code using case-insensitive `.zip` and `.uqm` matching, the subsystem shall remain compatible with that discovery and mount flow.
- **REQ-UIO-ARCHIVE-007** [engine-critical] When the engine enumerates `.rmp` resource index files through directory listing and match filtering in the proven startup use case, the subsystem shall provide matching behavior compatible with that startup path. *Note: This requirement covers the regex/match-filtering behavior itself, not a proof that combined STDIO+archive merged discovery is currently required for parity. See REQ-UIO-LIST-016 for the broader, currently unverified merged-listing-across-mount-types target, which remains compatibility-complete pending verification.*
- **REQ-UIO-ARCHIVE-008** [engine-critical] When archive content has been mounted and startup completes, resource consumers shall be able to read package content through the same exported APIs (`uio_fopen`, `uio_fread`, `uio_fseek`, `uio_ftell`, etc.) used for non-archive content.
- **REQ-UIO-ARCHIVE-009** [compatibility-complete] When a caller tests accessibility of archive-backed content via `uio_access`, the subsystem shall report archive entries as readable and existing, shall reject write and execute access, and shall report existence for directories synthesized from archive paths.
- **REQ-UIO-ARCHIVE-010** [compatibility-complete] When a caller queries the physical location of archive-backed content via `uio_getFileLocation`, the subsystem shall return successful location information sufficient to identify the owning mount and support legacy helper-stack consumers such as `uio_getStdioAccess`. *Note: Observed legacy helper behavior in `utils.c` calls `uio_getFileLocation()` before deciding whether direct stdio access is possible or whether a temporary-copy fallback is required. The exact path/string format returned for non-STDIO-backed content is still not fully specified by the current evidence and remains an open compatibility question, but outright `ENOTSUP` failure is inconsistent with the observed helper stack.*
- **REQ-UIO-ARCHIVE-011** [compatibility-complete] When a caller requests stdio access for archive-backed content via `uio_getStdioAccess`, the subsystem shall follow the observed legacy helper-stack behavior: use direct path access for STDIO-backed content and otherwise create a temporary copy whose stdio path is returned to the caller. *Note: Observed legacy `utils.c` behavior creates a temporary directory, copies the file there, and returns a stdio path when the owning filesystem type is not `uio_FSTYPE_STDIO`. Clean failure remains required when location discovery, temporary-directory creation, file copy, or temporary-path resolution fails.*

## FileBlock requirements

*Note: The FileBlock API has in-tree usage in the legacy ZIP implementation (`sc2/src/libs/uio/zip/zip.c`), which uses `uio_openFileBlock2`, `uio_copyFileBlock`, `uio_accessFileBlock`, and `uio_setFileBlockUsageHint` extensively. However, whether any engine-critical runtime path exercises FileBlock through the exported public API is not established by `initialstate.md`. If the Rust implementation provides archive support independently of the legacy FileBlock layering, these APIs remain compatibility-complete as exported symbols. If a future implementation preserves the legacy ZIP layering, FileBlock would need to be elevated accordingly. These requirements are classified as compatibility-complete unless proven engine-critical. Behavioral expectations below are proposed design based on API signatures and header comments rather than proven legacy contract, and should be treated as revisable until consumer usage is confirmed.*

- **REQ-UIO-FB-001** [compatibility-complete] When a caller opens a file block for a handle, the subsystem shall create a block object representing a contiguous readable region of that file.
- **REQ-UIO-FB-002** [compatibility-complete] When a caller opens a file block with an explicit offset and size, the subsystem shall create a block object constrained to that file subrange.
- **REQ-UIO-FB-003** [compatibility-complete] When a caller accesses a file block range, the subsystem shall return access to the requested bytes up to the available extent of the represented block.
- **REQ-UIO-FB-004** [compatibility-complete] When a caller copies bytes from a file block, the subsystem shall copy the requested bytes up to the available extent of the represented block.
- **REQ-UIO-FB-005** [compatibility-complete] When a caller provides file block usage hints, the subsystem should use those hints to improve access behavior when practical and shall otherwise safely preserve correctness.
- **REQ-UIO-FB-006** [compatibility-complete] When a caller closes a file block, the subsystem shall release any buffers, mappings, and ownership references held by that block.
- **REQ-UIO-FB-007** [compatibility-complete] While a file block remains live, the subsystem shall keep any dependent file-handle ownership needed for valid block access.

## Stdio access bridge requirements

*Note: The StdioAccess API has in-tree usage in legacy helper and debug code (`sc2/src/libs/uio/utils.c:166–210`, `sc2/src/libs/uio/debug.c:472`). However, whether any engine-critical runtime path depends on StdioAccess through the exported public API is not established by `initialstate.md`. These requirements are classified as compatibility-complete unless proven engine-critical, but they still describe supported in-tree exported surface and must not be treated as optional. Behavioral expectations below are proposed design based on API signatures and expected use cases rather than proven legacy contract, and should be treated as revisable until consumer usage is confirmed.*

- **REQ-UIO-STDIO-001** [compatibility-complete] When a caller requests stdio access for content already backed by a native filesystem path, the subsystem shall return a handle exposing that usable path.
- **REQ-UIO-STDIO-002** [compatibility-complete] When a caller requests stdio access for content not directly backed by a usable native filesystem path, the subsystem shall either provide a stronger documented success path for that backing type or fail cleanly if the required fallback cannot be completed. *Note: For archive-backed content, the observed legacy helper-stack behavior is to create a temporary copy and return its path rather than immediately fail with `ENOTSUP`. Other non-native-backed cases remain subject to their documented backing-type-specific contract.*
- **REQ-UIO-STDIO-003** [compatibility-complete] When temporary resources are required to satisfy a stdio access request, the subsystem shall manage those resources within the caller-provided context and shall clean them up on release.
- **REQ-UIO-STDIO-004** [compatibility-complete] When stdio access is released, the subsystem shall clean up any temporary resources that were created to satisfy that request when cleanup is possible.
- **REQ-UIO-STDIO-005** [compatibility-complete] While a stdio access handle remains live, the native path returned by that handle shall remain valid.
- **REQ-UIO-STDIO-006** [compatibility-complete] When a stdio access request fails, the subsystem shall not leak partially created temporary resources.

## Ownership and memory-management requirements

- **REQ-UIO-MEM-001** [engine-critical] When the subsystem successfully creates an object (repository, mount handle, directory handle, file handle, stream, directory list, file block, or stdio access handle), a single matching close or free call for that object shall release all owned resources without leaks or crashes.
- **REQ-UIO-MEM-002** [engine-critical] When an object's documented close or free function is called exactly once after successful creation, the subsystem shall release all memory and owned resources associated with that object.
- **REQ-UIO-MEM-003** [engine-critical] While an object remains owned by the caller, the subsystem shall keep all exposed internal pointers valid according to that object's public contract.
- **REQ-UIO-MEM-004** [engine-critical] When a stream is closed, the subsystem shall not leak stream-owned buffers. *Consumer impact: The current implementation acknowledges leaking the stream buffer on close. This must be fixed.*
- **REQ-UIO-MEM-005** [quality/cleanup] When a directory list is freed, the subsystem should be able to perform correct deallocation without relying on external side-channel registries that could become inconsistent. *Rationale: The observable requirement is leak-free deallocation (covered by REQ-UIO-MEM-002). This requirement expresses a preference for self-describing allocation strategies that reduce fragility. Any approach that achieves correct deallocation is acceptable; the preference for avoiding global side-channel registries is an implementation quality guideline.*
- **REQ-UIO-MEM-006** [quality/cleanup] When a mount or repository cleanup path runs, the subsystem should free mount-handle allocations that are no longer reachable. *Rationale: The observable failure mode is memory leaks during shutdown. This is internal hygiene, not a caller-visible contract.*
- **REQ-UIO-MEM-007** [compatibility-complete] When a cleanup path encounters null object pointers where null is a valid no-op input for the public API, the subsystem should treat the input safely.

## Error-handling requirements

- **REQ-UIO-ERR-001** [engine-critical] When a public UIO operation fails, the subsystem shall return the failure sentinel defined by that operation's public contract.
- **REQ-UIO-ERR-002** [engine-critical] When a public UIO operation fails for a reason that maps to engine-visible OS-style error reporting, the subsystem shall set `errno` to a meaningful value from the error categories defined in specification §6.2.1 before returning. *Consumer dependency: `sdluio.c` calls `strerror(errno)` after UIO failures. `options.c` also checks errno-based error messages.*
- **REQ-UIO-ERR-003** [compatibility-complete] When an operation fails after partially allocating internal resources, the subsystem shall clean up those partially allocated resources before returning failure.
- **REQ-UIO-ERR-004** [engine-critical] When an operation receives null pointers, invalid flags, or otherwise detectably invalid arguments, the subsystem shall return the documented failure sentinel, shall not crash or exhibit undefined behavior, and shall set `errno` to `EINVAL`.
- **REQ-UIO-ERR-005** [engine-critical] When a stream read returns zero items because of EOF rather than I/O failure, the subsystem shall preserve status information that allows consumers to distinguish EOF from error.
- **REQ-UIO-ERR-006** [engine-critical] When a stream read or write returns zero items because of I/O failure, the subsystem shall preserve status information that allows consumers to distinguish error from EOF.
- **REQ-UIO-ERR-007** [compatibility-complete] When an operation is unsupported for the resolved filesystem type or object state, the subsystem shall fail explicitly and shall set `errno` to `ENOTSUP` or an equivalent value rather than silently succeeding with incomplete behavior.
- **REQ-UIO-ERR-008** [engine-critical] If the subsystem encounters an internal failure while servicing an FFI call, then it shall convert that failure into API-level error reporting and shall not unwind across the foreign-language boundary.
- **REQ-UIO-ERR-009** [compatibility-complete] When a public UIO operation succeeds, the subsystem may leave `errno` in an unspecified state. Callers shall not rely on `errno` being preserved or cleared after successful calls.
- **REQ-UIO-ERR-010** [engine-critical] When a file or directory is not found during path resolution, the subsystem shall set `errno` to `ENOENT` before returning failure. *Consumer dependency: `options.c` and `sdluio.c` use `strerror(errno)` to report meaningful error messages; `ENOENT` is the expected code for missing-file diagnostics.*
- **REQ-UIO-ERR-011** [engine-critical] When a write, delete, or modify operation fails because the target mount is read-only or archive-backed, the subsystem shall set `errno` to `EROFS` or `EACCES` before returning failure.
- **REQ-UIO-ERR-012** [compatibility-complete] When an exported API is not yet implemented, the subsystem shall return the public failure sentinel for that API's return type, shall set `errno` to `ENOTSUP` or an equivalent meaningful value, and shall not return placeholder success objects (non-null handles, dummy pointers, or zero return codes) that defer failure to a later unrelated call site. *Rationale: Stubbed APIs that return fake success indicators cause callers to proceed as if the operation succeeded, producing mysterious downstream failures instead of immediate, diagnosable errors at the call site. This applies to all currently stubbed exported APIs including FileBlock, StdioAccess, and any other compatibility-complete APIs not yet functionally implemented.*

## Concurrency and thread-safety requirements

- **REQ-UIO-CONC-001** [engine-critical] When multiple threads perform independent UIO operations against the subsystem using separate handles, the subsystem shall preserve internal memory safety and object-lifetime correctness. *Consumer dependency: concurrent audio decoding threads resolve paths and read from separate file handles while the main thread is not mutating mounts.*
- **REQ-UIO-CONC-002** [engine-critical] While concurrent callers access the mount registry for path resolution reads, the subsystem shall prevent readers from observing torn mount-state updates. Mount-registry mutations shall be serialized.
- **REQ-UIO-CONC-003** [compatibility-complete] While concurrent callers operate on the same file handle or stream-backed underlying handle, the subsystem shall preserve handle integrity and shall avoid data races in internal state. *Note (provisional): The degree to which current consumers exercise same-handle concurrency is not fully established. `initialstate.md` notes possible concurrent audio decoding threads, but does not demonstrate same-handle sharing. Until same-handle concurrent usage is demonstrated, this is treated as a desired robustness guarantee rather than a proven legacy obligation.*
- **REQ-UIO-CONC-004** [compatibility-complete] When one thread unmounts content or closes a repository, callers shall not assume that previously resolvable paths remain valid unless the corresponding objects are independently kept alive by the public lifetime contract.
- **REQ-UIO-CONC-005** [quality/cleanup] When internal locking is used, the subsystem should minimize lock hold time for global mount state and should avoid holding global namespace locks across blocking file I/O when not required for correctness.

## Lifecycle and handle-validity requirements

- **REQ-UIO-LIFE-001** [engine-critical] When a repository is closed via `uio_closeRepository`, the subsystem shall unmount all mounts before freeing the repository. After return, the repository pointer shall be invalid.
- **REQ-UIO-LIFE-002** [engine-critical] When a mount is unmounted via `uio_unmountDir`, the mount handle shall become invalid and shall not be usable as a relative-placement anchor for subsequent mounts.
- **REQ-UIO-LIFE-003** [compatibility-complete] When a directory handle's underlying mount is unmounted while the handle remains live, the subsystem shall not crash or exhibit undefined behavior. The directory handle remains a valid allocated object until `uio_closeDir` is called. *Note (proposed safe-failure robustness target — not verified legacy behavior): Whether the legacy C implementation enforced clean-failure semantics for post-unmount directory-handle operations is unknown — see specification §5.8.1. The no-crash/no-UB objective is the hard requirement. Whether post-unmount operations return specific error codes or exhibit other defined functional behavior beyond the no-crash/no-UB guarantee is provisional and acceptance-test-defined pending legacy verification.*
- **REQ-UIO-LIFE-004** [compatibility-complete] When a file handle or stream remains open while its backing mount is unmounted, the subsystem shall preserve a minimum safety floor: subsequent operations through that already-open object shall not crash the process or produce undefined behavior. *Note (provisional): The exact post-unmount functional behavior for already-open file handles and streams is not yet established. Implementations may keep the object usable until close, or may cause subsequent operations to fail cleanly with documented error signaling, but they must not return misleading success. Legacy behavior for this case requires verification — see specification §5.8.1.*
- **REQ-UIO-LIFE-005** [compatibility-complete] When UQM's shutdown sequence violates the expected ordering (close files, then close dirs, then unmount, then close repository), the subsystem shall preserve that same no-crash/no-undefined-behavior safety floor. Exact functional behavior beyond that safety floor remains provisional pending legacy verification.

## Integration and compatibility requirements

- **REQ-UIO-INT-001** [engine-critical] When the engine startup code mounts base content, config content, addon content, and package archives through the established UIO API, the subsystem shall support that orchestration model without requiring startup-policy ownership to move into the UIO subsystem.
- **REQ-UIO-INT-002** [engine-critical] When engine code stores UIO objects in existing globals such as content, config, save, melee, and content-mount handles, the subsystem shall remain compatible with that external ownership model.
- **REQ-UIO-INT-003** [engine-critical] When SDL RWops integration uses the UIO stream API, the subsystem shall provide stream semantics sufficient for SDL callers to distinguish successful reads, EOF, and error conditions. *Consumer dependency: `sdluio.c` checks `uio_ferror(stream)` after zero-byte reads and reports via `SDL_SetError` with `strerror(errno)`.*
- **REQ-UIO-INT-004** [engine-critical] When Rust sound, resource, and other engine subsystems call the exported UIO C ABI, the subsystem shall preserve ABI compatibility for the public data structures and function signatures they consume.
- **REQ-UIO-INT-005** [engine-critical] When the build is configured to use Rust UIO, the subsystem shall provide externally observable behavior that matches the documented engine-critical and supported compatibility-complete obligations for the enabled build surface, and it shall not hide unsupported behavior behind compatibility shims or mixed-language boundaries by pretending to succeed where the documented contract is not met. *Note: This requirement is about correctness relative to the documented tiering model. It does not by itself forbid compatibility shims such as `uio_fread` when those shims preserve correct public behavior, and it does not convert every port-completeness/cleanup goal into an engine-critical parity gate.*
- **REQ-UIO-INT-006** [compatibility-complete] When consumers compile against the existing UIO public headers, the subsystem shall preserve the observable contract implied by those headers for supported functionality.
- **REQ-UIO-INT-007** [compatibility-complete] When memory-debug or diagnostic tooling invokes UIO debug-print entry points, the subsystem shall provide those entry points with behavior adequate for tooling integration. *Note: No proven runtime-critical dependency on these entry points has been established.*

## FFI robustness requirements

- **REQ-UIO-FFI-001** [engine-critical] When a public UIO function receives a null pointer argument, the subsystem shall return the documented failure sentinel and shall not crash. When a public UIO function receives a non-null but invalid pointer that cannot be reliably detected without dereference, the subsystem shall fail safely on a best-effort basis and shall not exhibit undefined behavior for detectable cases.
- **REQ-UIO-FFI-002** [engine-critical] When a foreign caller passes invalid input that cannot be safely serviced, the subsystem shall fail the call without crashing where reasonably possible.
- **REQ-UIO-FFI-003** [engine-critical] If internal implementation code panics or otherwise raises a language-native exception while servicing a foreign call, then the subsystem shall catch or translate that failure before the FFI boundary is crossed.
- **REQ-UIO-FFI-004** [engine-critical] While public FFI-visible structures are shared with existing consumers, the subsystem shall preserve the expected ABI layout and field semantics required by those consumers.

## Non-goals and boundary requirements

- **REQ-UIO-BOUND-001** [engine-critical] When startup code chooses what to mount and in which order, the subsystem shall not assume or depend on specific mount-order decisions made at the startup-policy level. Mount-order policy remains the responsibility of the external startup code.
- **REQ-UIO-BOUND-002** [engine-critical] When addon discovery, package discovery, or resource-index loading logic is performed by external engine code, the subsystem shall support that logic through mount, listing, and file APIs but shall not require UIO to own those policies.
- **REQ-UIO-BOUND-003** [engine-critical] When external adapters such as SDL RWops wrappers are used, the subsystem shall serve as the underlying I/O provider and shall not require those adapters to adopt implementation-specific knowledge beyond the public UIO contract.
