## Gap 1 — Zero-size allocation OOM paths unchecked

### Verdict
**CONFIRMED with revised requirement mapping**

### Line verification
The cited line numbers match the current file exactly:
- `rust_hmalloc` zero-size path: lines 10-12
- `rust_hcalloc` zero-size path: lines 42-46
- `rust_hrealloc` zero-size path: lines 64-69

### Evidence from code
From `rust/src/memory.rs`:

```rust
pub unsafe extern "C" fn rust_hmalloc(size: usize) -> *mut c_void {
    if size == 0 {
        // Return a non-null pointer for zero-size allocation
        return libc::malloc(1);
    }
```

```rust
pub unsafe extern "C" fn rust_hcalloc(size: usize) -> *mut c_void {
    if size == 0 {
        // Return a non-null pointer for zero-size allocation
        let ptr = libc::malloc(1);
        libc::memset(ptr, 0, 1);
        return ptr;
    }
```

```rust
pub unsafe extern "C" fn rust_hrealloc(ptr: *mut c_void, size: usize) -> *mut c_void {
    if size == 0 {
        // If new size is 0, free the pointer and return a minimal allocation
        if !ptr.is_null() {
            libc::free(ptr);
        }
        return libc::malloc(1);
    }
```

Positive-size paths do abort on null:

```rust
let ptr = libc::malloc(size);
if ptr.is_null() {
    log_add(LogLevel::Fatal, "HMalloc() FATAL: out of memory.");
    std::process::abort();
}
```

```rust
let ptr = libc::malloc(size);
if ptr.is_null() {
    log_add(LogLevel::Fatal, "HCalloc() FATAL: out of memory.");
    std::process::abort();
}
```

```rust
let new_ptr = libc::realloc(ptr, size);
if new_ptr.is_null() && size > 0 {
    log_add(LogLevel::Fatal, "HRealloc() FATAL: out of memory.");
    std::process::abort();
}
```

### Accuracy of description
The gap description is accurate.
- `rust_hmalloc(0)` returns `libc::malloc(1)` without a null check.
- `rust_hcalloc(0)` uses `malloc(1)` and then immediately calls `memset(ptr, 0, 1)` without verifying `ptr` is non-null.
- `rust_hrealloc(_, 0)` returns `malloc(1)` without a null check.

This does violate the spec’s “returns non-null or does not return” contract for zero-size normalization fallback.

### Specification verification
The cited specification references are correct:
- Specification §4.1 lines 113-115 explicitly require fatal behavior if the 1-byte fallback fails.
- Specification §6.3 lines 211-214 explicitly extends OOM detection to the internal 1-byte zero-size normalization allocations.

### Requirements verification
The referenced requirement IDs exist, but the mapping is incomplete.
- Present and applicable:
  - `REQ-MEM-OOM-001` at requirements line 85
  - `REQ-MEM-ZERO-001` at line 63
  - `REQ-MEM-OOM-003` at line 95
- Additional directly applicable requirement missed by the analysis:
  - `REQ-MEM-ZERO-002` at line 68: `HCalloc(0)` must not perform invalid memory operations, including invalid zero-fill behavior on a null pointer.

### Correction
Gap 1 should also cite **REQ-MEM-ZERO-002**, because `rust_hcalloc(0)` currently performs:

```rust
let ptr = libc::malloc(1);
libc::memset(ptr, 0, 1);
```

which would be invalid if `ptr` were null.

## Gap 2 — `copy_argv_to_c` uses wrong deallocator for CString pointers

### Verdict
**CONFIRMED**

### Line verification
The cited line numbers match the current file exactly:
- Error-path cleanup: lines 122-129, with the wrong free at line 126
- Test cleanup: lines 269-279, with the wrong free at line 276

### Evidence from code
Error path in `copy_argv_to_c`:

```rust
let array_ptr = rust_hmalloc(std::mem::size_of::<*mut i8>() * (argv.len() + 1)) as *mut *mut i8;
if array_ptr.is_null() {
    // Clean up the allocated strings
    for ptr in &c_strings {
        if !ptr.is_null() {
            libc::free(*ptr as *mut c_void);
        }
    }
    panic!("Failed to allocate argv array");
}
```

Test cleanup:

```rust
loop {
    let ptr = *array_ptr.add(i);
    if ptr.is_null() {
        break;
    }
    libc::free(ptr as *mut c_void);
    i += 1;
}
rust_hfree(array_ptr as *mut c_void);
```

CString creation uses `into_raw()`:

```rust
let c_string = CString::new(arg.as_str()).expect("Failed to convert argument to C string");
c_strings.push(c_string.into_raw());
```

### Accuracy of description
The gap description is accurate.

Pointers produced by `CString::into_raw()` must be reclaimed with `CString::from_raw()`, not `libc::free()`. The current code violates the allocator-family contract described in the specification.

The analysis is also correct that the `array_ptr.is_null()` branch is effectively unreachable under the current `rust_hmalloc` implementation for positive-size requests, because `rust_hmalloc` aborts on OOM instead of returning null:

```rust
let ptr = libc::malloc(size);
if ptr.is_null() {
    log_add(LogLevel::Fatal, "HMalloc() FATAL: out of memory.");
    std::process::abort();
}
```

That said, the branch does still exist in source and currently contains both:
- a wrong deallocator call, and
- a panic across a Rust-only boundary.

### Specification verification
The cited specification reference is correct:
- Appendix A.3 lines 409-413 explicitly state that `CString::into_raw()` allocations must be reclaimed with `CString::from_raw()`, never via `libc::free` or `HFree`.

### Requirements verification
The referenced requirement IDs exist:
- `REQ-MEM-OWN-006` at requirements line 174
- `REQ-MEM-INT-008` at line 221

These references are reasonable and supported.

## Additional Gaps Found

### Additional Gap A — `copy_argv_to_c` documentation/return semantics do not match Specification Appendix A.2

#### Verdict
**MISSED BY ORIGINAL ANALYSIS**

#### Evidence from code
The function returns both the array pointer and the vector of raw string pointers:

```rust
pub unsafe fn copy_argv_to_c(argv: &[String]) -> (*mut *mut i8, Vec<*mut i8>) {
```

and ends with:

```rust
(array_ptr, c_strings)
```

But the in-code safety docs say only:

```rust
/// This function allocates memory that must be freed with HFree
```

That statement is incomplete and misleading because only the pointer array is `rust_hmalloc`/`HFree`-owned; the individual strings are Rust `CString::into_raw()` allocations and must be reclaimed with `CString::from_raw()`.

#### Specification verification
Specification Appendix A.2/A.3 states:
- A.2 lines 404-407: function allocates a pointer array via `rust_hmalloc`, returns the array pointer and a `Vec` of the individual string pointers.
- A.3 lines 411-413: array must be freed via `rust_hfree`; string pointers must be reclaimed with `CString::from_raw()`.

#### Why this is a real gap
The implementation behavior matches the spec return shape, but the Rustdoc contract in the source file does not. A caller reading only the source docs could reasonably infer that all allocated memory should be freed with `HFree`, which would be wrong for the string pointers.

This is closely related to Gap 2 but distinct: Gap 2 is about incorrect cleanup behavior; this additional gap is about incorrect/incomplete API documentation for the helper function.
