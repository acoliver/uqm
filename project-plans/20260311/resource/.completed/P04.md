unsafe extern "C" fn unknownres_load_fun(descriptor: *const c_char, data: *mut ResourceData) {

### 3. dispatch.rs — Value-Type get_resource Path

**File**: `rust/src/resource/dispatch.rs`  
**Function**: `get_resource`  
**Lines**: Added value-type detection and return path before heap-type lazy loading

**Implementation**:
```rust
// @plan PLAN-20260314-RESOURCE.P04
// @requirement REQ-RES-LOAD-011
// Check if this is a value type (no free_fun)
let is_value_type = self
    .type_registry
    .lookup(&desc.type_handler_key)
    .map(|h| h.free_fun.is_none())
    .unwrap_or(false);

if is_value_type {
    // Value type: return str_ptr if non-null, else num as pointer
    desc.refcount += 1;
    let str_ptr = unsafe { desc.data.str_ptr };
    if !str_ptr.is_null() {
        return str_ptr as *mut c_void;
    } else {
        return unsafe { desc.data.num } as *mut c_void;
    }
}

// Heap type: lazy load if not yet loaded (ptr is null)
[existing heap-type logic unchanged]
```

**Rationale**:
- Value types (STRING, INT32, BOOLEAN, COLOR, UNKNOWNRES) return their stored value directly
- STRING and UNKNOWNRES return `str_ptr` (descriptor pointer)
- INT32, BOOLEAN, COLOR return `num` cast as pointer
- Refcount is incremented for all successful returns
- Heap types continue to use existing lazy-load logic

## Implementation Notes

### Design Decisions

1. **Separate load functions**: Created both `unknownres_load_fun` (in ffi_bridge.rs, for production) and `unknownres_load_wrapper` (in dispatch.rs tests, for test isolation)

2. **Value-type detection strategy**: Check `free_fun.is_none()` in `get_resource()` to determine if entry is a value type, avoiding duplication of value-type list

3. **Return priority for value types**: Return `str_ptr` if non-null, else `num` as pointer. This handles STRING/UNKNOWNRES (str_ptr) and INT32/BOOLEAN/COLOR (num) correctly

4. **Handler key fix**: Changed eager-load lookup from `type_name` to `handler_key` to ensure UNKNOWNRES fallback entries actually call the load function

### Edge Cases Handled

1. **Null str_ptr for numeric value types**: Falls back to returning `num` as pointer
2. **Refcount for value types**: Incremented before return (not after load like heap types)
3. **Unknown fallback during eager load**: Uses `handler_key` ("UNKNOWNRES") not original type name for lookup
