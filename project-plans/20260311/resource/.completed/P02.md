### PC-3: get_resource Value-Type Path

**Status**: [OK] CORRECT with minor clarification needed

Lines 76-80 implement value-type return logic:

```text
76:     IF entry.data.str_ptr IS NOT null THEN
77:       RETURN Some(entry.data.str_ptr as *mut c_void)
78:     ELSE
79:       RETURN Some(entry.data.num as *mut c_void)
80:     END IF
```

**Logic**: Discriminate based on `str_ptr != null` → return string, else numeric.

**Correctness Check**:
- STRING: loads descriptor pointer into `str_ptr` [OK]
- UNKNOWNRES: loads descriptor pointer into `str_ptr` [OK]
- INT32/BOOLEAN/COLOR: load into `num` [OK]

**Edge case**: What if BOTH `str_ptr` and `num` are null/zero?

From P01 GAP-4 analysis:
> For value types, `data` is stored in either:
> - `data.num`, or
> - `data.str_ptr`

For value types, the data is **always populated during `process_resource_desc`**, so this edge case should never occur for properly loaded entries.

**Recommendation**: Add assertion or early return for "both null" case:

```text
71:     // --- Value type: already populated ---
72:     IF entry.data.str_ptr IS null AND entry.data.num == 0 THEN
73:       WARN "Value-type entry has no data: {key}"
74:       RETURN None
75:     END IF
76:     entry.refcount += 1
```
