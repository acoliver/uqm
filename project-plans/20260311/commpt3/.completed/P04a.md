## Semantic Negative-Proof Confirmations

### C stub bodies extracted by tests (verbatim):
Both `c_SetColorMapFromCommData` and `c_PlayAlienMusic` have bodies of exactly:
```
{
	/* stub — impl in P05 */
}
```

Confirmed by panic messages from the expected-fail tests:
- `stub body: "{\n\t/* stub — impl in P05 */\n}"`

### Tests use function-body-aware extraction (`extract_c_fn_body`):
- Scans for the function definition signature (not in comment, not a declaration)
- Skips matches where a `;` appears between function name and `{` (declarations)
- Returns only the brace-balanced body
- Doc-comment lines above the stub (which contain `CommData.AlienSong`, `CommData.AlienColorMap` as documentation) are **outside** the extracted body

Confirmed: the doc-comments at lines 1158–1162 (PlayAlienMusic) and 1196–1200 (SetColorMapFromCommData) mention CommData fields, but the body extractor returns only `{\n\t/* stub — impl in P05 */\n}`. The tests therefore correctly fail — no false positive from doc-comments.

### Expected-fail tests fail for the right reason:
- `verify_c_bridge_reads_commdata_colormap`: fails because stub body has no `CommData.AlienColorMap`
- `verify_c_bridge_null_guard_colormap`: fails because stub body has no `== 0`, `!= 0`, `== NULL`, `!= NULL`, `if (!`, or `if (cmap`
- `verify_c_music_reads_commdata`: fails because stub body has no `CommData.AlienSong`
- `verify_c_bridge_functions_exist_with_impl`: fails because stub body has zero non-blank, non-brace, non-comment lines

### Trivial-wrong-implementation would still fail:
A hardcoded constant (e.g., `SetColorMap(some_constant)`) in the body would:
- Pass `verify_c_bridge_functions_exist_with_impl` (non-empty)
- Still FAIL `verify_c_bridge_reads_commdata_colormap` (no `CommData.AlienColorMap`)
- Still FAIL `verify_c_bridge_null_guard_colormap` (no guard)
→ Tests are non-vacuous.
