## Open Questions from Specification §17

### Q1: `uio_Stream` layout ABI visibility
**Resolution: field order is not frozen for external C consumers, but must remain compatible with internal C modules compiled with `uio_INTERNAL` if they ever see the struct; current known Rust-UIO call sites do not directly access fields.**

Evidence:
- `uiostream.h:75-97` defines the internal C layout.
- `sdluio.c:72-149` only stores/passes `uio_Stream *`; it never dereferences fields.
- Search results show direct field access exists in `uiostream.c` (`stream->status`, `stream->operation`, etc.), but that is the C implementation file for streams, not a separate Rust-UIO consumer.
- Under `USE_RUST_UIO`, `uio_fread` is redirected via shim (`uio_fread_shim.c`), reducing direct dependence on stream internals from general callers.
- Rust struct layout currently mirrors the C field order semantically at `uio_bridge.rs:269-278`, though Rust names differ (`data_start`, `data_end`, `open_flags`).

Carry-forward requirement:
- Later phases should preserve current `repr(C)` field order unless and until build evidence proves no `uio_INTERNAL` consumer of Rust `uio_Stream` layout remains.

### Q2: AutoMount parity requirement
**Resolution: AutoMount can be deferred for engine/runtime paths, but only with explicit documentation that current production callers pass sentinel arrays `{ NULL }`; feature-complete parity is still missing for debug/test scenarios.**

Evidence:
- `options.c:207`, `333`, `369`, `466` define `static uio_AutoMount *autoMount[] = { NULL };` and pass them to `uio_mountDir`.
- `temp.c:115` and `gfxload.c:43` also pass `{ NULL }` arrays.
- Non-trivial automount rules appear in `libs/uio/debug.c:149-156` (`autoMountZip`, non-NULL array entries), which is not evidence of normal engine startup but does prove feature exists in C.
- Rust `uio_mountDir` ignores `_autoMount` entirely at `uio_bridge.rs:1461`.

Plan branch implication:
- For mainline FILE-IO plan, AutoMount need not block early phases if scope is runtime parity for existing engine callers.
- Final plan notes must explicitly record that debug/test-only non-NULL AutoMount behavior remains unsupported unless Phase 07/09 is expanded.

### Q3: Temp-directory mounting
**Resolution: process-level temp-directory mounting is required by current engine startup and cannot be deferred.**

Evidence:
- `uqm.c:411` calls `initTempDir()` and `uqm.c:499` calls `unInitTempDir()`.
- `libs/file/temp.c:114-130` mounts a temp directory into repository namespace at `"/tmp/"` using `uio_mountDir(...)`, then opens `tempDir = uio_openDir(repository, "/tmp", 0);`.
- This is distinct from per-handle `uio_getStdioAccess(..., tempDir)` behavior in `utils.c:165-323`.

Plan branch implication:
- Temp-mount support must be reserved in later phases; it is not optional.

## Additional Preflight Decisions

### Regex compatibility decision
**Decision: exact POSIX ERE parity should be treated as required until narrower evidence exists; do not assume Rust’s eventual regex engine can be weaker.**

Evidence:
- Production code uses regex matching through `uio_getDirList(..., match_MATCH_REGEX)` in `options.c:469-470` for ZIP/UQM discovery and `options.c:496-497` for `.rmp` discovery.
- C matcher explicitly supports multiple match modes in `libs/uio/match.c`, including `match_MATCH_REGEX`.
- No Rust regex crate is present yet, and no preflight evidence proves a narrower pattern subset beyond the currently observed simple case-insensitive extension regexes.

Practical note:
- Observed current production patterns are simple extension regexes, so implementation can start with audited subset support, but the phase documentation must not overclaim generic regex parity without further audit.

### ZIP / FileBlock dependency rationale
**Decision: Phase 09 should treat public FileBlock as a real dependency for C ZIP parity; direct Rust ZIP reading would not eliminate required ABI work.**

Evidence:
- C ZIP implementation heavily depends on FileBlock APIs (`zip.c` call sites listed above, especially `uio_openFileBlock2`, `uio_accessFileBlock`, `uio_copyFileBlock`).
- Rust FileBlock exports are pure stubs (`uio_bridge.rs:911-966`).
- No Rust ZIP/archive dependency exists in `Cargo.toml`.
- Even if Rust ZIP reading were added later, the public ABI still exposes FileBlock in `fileblock.h:73-84`, and existing C ZIP code would remain broken until FileBlock is implemented or ZIP code is rewritten.

### Panic-safety boundary strategy
**Decision: add typed internal wrapper helpers per return family and migrate all exported `extern "C"` entry points behind them; current coverage is far from sufficient.**

Evidence:
- `uio_bridge.rs` has 77 `#[no_mangle]` exports.
- Only `uio_fgets` wraps the full body in `catch_unwind` (`613-624`), and `rust_uio_fread` only wraps one dereference (`1890-1898`).
- Most exports perform raw pointer dereferences or heap operations without unwind containment.

Required return-family strategy:
- pointer returns -> `NULL`
- signed integer / ssize_t / c_long returns -> `-1` (or `EOF` where stdio semantics require)
- size_t returns -> `0`
- void returns -> swallow panic after logging

Proof strategy for later phases:
- introduce small helper wrappers/macros around closures for each return family,
- convert all `extern "C"` exports to use them,
- add panic-injection tests on representative functions from each family to prove no unwind crosses FFI.

### Post-unmount cleanup safety audit
- FAIL — cleanup after mount removal is currently unsafe/incomplete.

Evidence:
- Relevant cleanup paths identified:
  - `uio_close` at `uio_bridge.rs:1643-1649`
  - `uio_fclose` at `1825-1841`
  - `uio_closeDir` at `1437-1450`
  - `uio_releaseStdioAccess` at `uio_bridge.rs:1205-1210` (stub) versus real cleanup semantics in `utils.c:296-318`
- `uio_unmountDir`/`uio_unmountAllDirs` only remove registry entries and free mount handles (`529-552`); they do not coordinate with outstanding handles.
- Rust `uio_getStdioAccess`/`uio_releaseStdioAccess` are stubs and do not preserve temp-root/temp-dir/file cleanup state required by C `utils.c`.
- Existing dir/file handles store resolved filesystem paths directly, so some operations may continue after unmount by accident, but this behavior is undocumented and not proven safe or semantically correct.

## Blocking Issues
1. `uio_openFileBlock2` Rust ABI is wrong: Rust signature is `(_handle, _flags)` at `uio_bridge.rs:919-922`, but C header requires `(handle, offset, size)` in `fileblock.h:74-75`.

3. `uio_getStdioAccess` Rust ABI is wrong: Rust takes 3 params at `uio_bridge.rs:1194-1199`, but C header requires 4 params including `tempDir` in `utils.h:34-35`.
4. `uio_vfprintf` is a stub, but netplay callers exist.
5. `uio_vasprintf` / `uio_asprintf` are not exported by Rust but are required by `uiostream.c`.
6. FileBlock functionality is stubbed while ZIP code has many live callers.
7. Full `sc2` build command in plan is currently not executable because `/Users/acoliver/projects/uqm/sc2` has no makefile at its top level.
8. Temp-directory repository mount support is required by engine startup (Q3), so deferral would be incorrect.
