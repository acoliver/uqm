## Purpose

Verify the cross-subsystem adoption of the pending-completion handshake after the provider-side implementation is in place. This phase is intentionally dedicated to comm integration because provider-only tests are not enough to prove the consumer contract.

## Requirements Implemented (Expanded)

### Comm integration for pending completion
**Requirement text**: The audio-heart subsystem shall preserve the externally visible behavior required by the comm subsystem for track playback, subtitle polling, seeking, and phrase progression, with main-thread-owned advancement behavior where the contract requires it.

Behavior contract:
- GIVEN: a phrase completion becomes pending
- WHEN: the comm-side main-thread loop polls and processes it
- THEN: callback execution occurs on the consumer/main-thread path and track advancement is committed coherently

- GIVEN: a completion is already pending
- WHEN: a seek request arrives
- THEN: the system follows the deferred/defined behavior from P09 without corrupting track state

- GIVEN: track advancement has not yet been committed
- WHEN: `PlayingTrack()` is queried
- THEN: it remains on the pre-commit phrase until commit occurs

- GIVEN: StopTrack is called with pending completion present
- WHEN: the comm loop continues polling
- THEN: no stale completion is delivered and pending state is cleared

Why it matters:
- This is one of the most behaviorally sensitive changes in the subsystem
- It crosses subsystem boundaries and directly affects comm subtitle progression and phrase control
- Generic "game launch" verification is not sufficient to prove this contract

## Implementation / Verification Tasks

### Comm-side verification targets
Use focused tests, harnesses, or scripted/manual probes to validate:
1. **Main-thread callback execution path**
   - prove the consumer executes claimed completion work on the intended thread/context
2. **Subtitle progression at phrase boundaries**
   - subtitle state changes only when the pending completion is claimed/committed according to the designed contract
3. **Seek while completion pending**
   - verify the defined defer/ignore behavior from P09
4. **`PlayingTrack()` pre-commit behavior**
   - remains nonzero / remains on the current phrase until commit occurs
5. **Stop clears pending completion**
   - no stale completion delivered after stop
6. **Pointer-identity-compatible subtitle polling still works in comm**
   - current subtitle pointer behavior remains coherent through phrase advancement
