## ZIP Archive Support Features

### Mount-Time Indexing
- ZIP archives are fully indexed when mounted via `uio_mountDir(..., UIO_FSTYPE_ZIP, ...)`
- Central directory is parsed and all entries are catalogued
- Mount fails with `errno=EIO` on corrupt or unreadable archives
- No residual registry entry on mount failure

### Path Normalization
- Leading "/" and "\" are stripped
- Backslashes converted to forward slashes
- Trailing "/" on directory entries is stripped
- Case-sensitive lookup (no case folding)

### Duplicate Entry Handling
- Last central directory entry wins when paths normalize to the same name
- Explicitly implemented per specification

### Synthetic Directories
- Implied parent directories are created (e.g., "a/b/c.txt" creates "a" and "a/b")
- Synthetic directories participate in listings and `uio_stat()`

### File Operations
- **uio_getDirList**: Includes ZIP entries in cross-mount listings
- **uio_access**: F_OK/R_OK succeed, W_OK fails with EACCES for ZIP entries
- **uio_stat**: Returns uncompressed size and correct mode bits (read-only)
- **uio_open/uio_fopen**: Opens ZIP entries for streaming decompression
- **uio_read**: Transparent decompression with CRC validation
- **Write operations**: Fail with EACCES (archives are read-only)

## Architecture Decisions

### Handle Design
Changed `uio_Handle` from a type alias to `Mutex<File>` to an enum-based approach:
```rust
pub enum uio_HandleInner {
    File(std::fs::File),
    ZipEntry(crate::io::zip_reader::ZipEntryReader),
}
```

This allows transparent handling of both regular files and ZIP entries through the same handle type while maintaining the C ABI via opaque pointers.

### Mount Registry Integration
ZIP indices are stored as `Arc<ZipIndex>` in `MountInfo`, allowing:
- Cheap cloning when resolving paths
- Safe concurrent access from multiple threads
- Automatic cleanup when mount is removed

### Performance Considerations
- **Eager indexing**: ZIP archives are fully indexed at mount time
- **No caching**: Individual file reads re-decompress on each access
- **Seek penalty**: Each seek in a ZIP entry requires re-reading from the beginning

Future optimization opportunities:
- LRU cache for frequently-accessed entries
- Buffered decompression state
- Mmap for uncompressed (stored) entries

### FileBlock vs Direct Reader Rationale
Chose direct `zip` crate integration over FileBlock because:
1. ZIP central directory requires random access for efficient parsing
2. FileBlock adds unnecessary indirection for archive formats
3. Direct integration allows better error handling and validation
4. Compression/decompression state management is cleaner without FileBlock wrapper
