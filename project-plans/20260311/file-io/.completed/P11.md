## Key Implementation Details

### Initialization Flag
- Used `AtomicBool` with `Ordering::SeqCst` for thread-safe initialization tracking
- Supports idempotent initialization (multiple calls safe)
- Supports safe uninitialization without prior initialization

### Registry Cleanup
- `uio_unInit()` clears both mount registry and buffer size registry
- Resets mount ID counter to initial state
- Leaves registries in clean state ready for re-initialization

### Post-Unmount Safety
All cleanup functions verified to be safe after mount removal because:
1. **uio_Handle**: Owns its file/ZIP entry reader directly, no mount dependency
2. **uio_Stream**: Owns its buffer and handle, both freed via buffer registry and Box::from_raw
3. **uio_DirHandle**: Uses refcounting, freed via Box::from_raw when refcount reaches 0
4. **uio_StdioAccessHandle**: Owns temp file path and cleanup state, manages temp files independently
5. **uio_DirList**: Buffer size tracked in side-channel registry, names array freed via Box

### Concurrency Safety Review
- Mount registry operations use Mutex for exclusive access during topology changes
- Initialization flag uses AtomicBool for lock-free state checks
- Individual handle operations don't require mount registry access for cleanup
- Buffer size registry uses Mutex for thread-safe allocation tracking

## Notes

### Design Decisions
1. **Side-channel buffer registry**: Required because C `uio_DirList` and `uio_Stream` structs cannot be modified to store buffer sizes. Used `HashMap<usize, BufferSizeEntry>` keyed by buffer pointer address.

2. **Idempotent initialization**: `uio_init()` checks initialization flag and skips if already initialized. This makes it safe for C code to call multiple times.

3. **Safe uninitialization**: `uio_unInit()` works even if `uio_init()` was never called, clearing state regardless. This prevents issues during error paths.

4. **Independent resource ownership**: All handle types (Handle, Stream, DirHandle, StdioAccess, DirList) own their resources directly and don't depend on mount registry for cleanup. This is the key guarantee for REQ-FIO-POST-UNMOUNT-CLEANUP.
