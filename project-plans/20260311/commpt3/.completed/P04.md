## Negative-Proof Rationale

The C structural tests (4-7) are function-body-aware: they use `extract_c_fn_body()` which extracts only the brace-balanced body of each function. This prevents false positives from the doc-comment lines above each stub, which mention `CommData.AlienColorMap` and `CommData.AlienSong` by name.

**Proof the tests are not vacuously passing:**
- All 4 C structural tests FAIL against the current P03 stubs (confirmed above).
- The stub body extracted by `extract_c_fn_body` is literally `{\n\t/* stub — impl in P05 */\n}` — confirmed in test failure messages.
- If `c_SetColorMapFromCommData` body were changed to `SetColorMap(0)` (wrong — no CommData read), `verify_c_bridge_reads_commdata_colormap` would still FAIL (no `CommData.AlienColorMap`).
- If `c_PlayAlienMusic` body were changed to `PlayMusic(NULL, 0, 0)` (wrong args), `verify_c_music_reads_commdata` would still FAIL (no `CommData.AlienSong`).
- Only a correct P05 implementation (reading `CommData.AlienColorMap`/`CommData.AlienSong` with proper null guards) will make all 4 tests pass.

**Plan grep checks note:** The plan's `grep -A5` commands return false positives because they include doc-comment lines above the stubs that name `CommData.AlienColorMap` and `CommData.AlienSong`. The Rust tests are more rigorous (function-body-aware) and correctly FAIL. This is a known limitation of the plan's grep approach, not a defect in the tests.
