### 2. Implementation Changes

**File:** `rust/src/resource/ffi_bridge.rs` (lines ~810-855)

**Before:**
- Used `string_cache` HashMap to cache CString values
- Returned `string_cache` entry pointer for any entry type
- Returned null for missing keys

**After:**
- Removed `string_cache` and related caching logic
- Added `static EMPTY: &[u8] = b"\0"` for empty string returns
- Early returns for:
  - Null key → empty string
  - Invalid UTF-8 → empty string
  - Uninitialized state → empty string
  - Missing key → empty string
  - Non-STRING type → empty string (explicit `entry.res_type != "STRING"` check)
  - Null `str_ptr` → empty string
- Returns `entry.data.str_ptr` only for STRING entries with non-null str_ptr

**Key behaviors implemented:**
- [OK] Never returns null (all error paths return EMPTY)
- [OK] Type check: only STRING entries return actual value
- [OK] INT32/BOOLEAN/COLOR/UNKNOWNRES entries return empty string
- [OK] Missing keys return empty string
- [OK] Null key input returns empty string

## Notes

1. **Design choice:** Removed `string_cache` HashMap because:
   - STRING entries store descriptor in `fname` field, not `data.str_ptr`
   - `data.str_ptr` for STRING is set by the load function and points to the CString in `fname_cstring`
   - Caching layer was redundant and introduced complexity
   - Direct return of `data.str_ptr` is safe because it's owned by the ResourceDesc entry

2. **Behavior change from C:** The new implementation is **safer** than C:
   - C could return null (crashes on `strlen()`)
   - Rust returns empty string (safe for all C callers)

3. **Test pattern established:** The 6 P05 tests establish a pattern for future res_Get* parity implementations
