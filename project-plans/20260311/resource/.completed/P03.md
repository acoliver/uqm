1. **test_unknownres_registered_as_value_type** - Gap: UNKNOWNRES has no load_fun
2. **test_process_resource_desc_unknown_type_stores_as_value** - Gap: No eager load for unknown types

### Unexpected Passes (2 tests)
Tests for STRING/INT32 value types passed due to union field aliasing:
- **test_get_resource_value_type_string_returns_str_ptr** - `ptr` and `str_ptr` are union aliases
- **test_get_resource_value_type_int_returns_num_as_ptr** - `ptr` and `num` read same memory

**Analysis**: These tests still serve their purpose for the GREEN phase. While the current code accidentally works due to union layout, the implementation doesn't *intentionally* distinguish value types from heap types. The real gaps are:
1. UNKNOWNRES lacks a load function
2. Unknown type fallback doesn't mark entries as value types
3. Eager loading uses wrong key (type_name instead of handler_key)

The GREEN phase will add explicit value-type handling, making the behavior intentional rather than accidental.

## Unexpected Findings

### Union Aliasing Masks Value-Type Gaps

The `ResourceData` union layout causes `ptr`, `str_ptr`, and `num` to share memory:

```rust
#[repr(C)]
pub union ResourceData {
    pub num: u32,
    pub ptr: *mut c_void,
    pub str_ptr: *const c_char,
}
```

**Consequence**: When value-type loaders set `str_ptr` or `num`, the current heap-centric `get_resource()` code that checks `data.ptr` accidentally reads the correct value.

**Why tests still valid**:
1. The behavior is undefined (depends on union field order)
2. The code doesn't *intentionally* handle value types
3. UNKNOWNRES still fails (no load function at all)
4. GREEN phase will add explicit value-type handling

**Verification**: Phase P04 (impl) will add explicit type discrimination and the tests will continue to pass with intentional logic.
