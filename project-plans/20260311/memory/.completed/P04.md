## Changes Made

### 1. Updated copy_argv_to_c documentation (lines 116-125)

Replaced the generic doc comment with detailed allocator-family ownership documentation:

```rust
/// Helper function to allocate memory for a C-compatible array of strings.
///
/// Returns a tuple of (array_ptr, string_ptrs):
/// - `array_ptr`: The null-terminated array of pointers. Must be freed via `rust_hfree`.
/// - `string_ptrs`: The individual C string pointers. Each must be reclaimed via
///   `CString::from_raw()`, NOT via `libc::free()` or `rust_hfree`, because they were
///   created by `CString::into_raw()` and are owned by the Rust allocator.
///
/// # Safety
/// The caller must ensure proper cleanup using the correct deallocator for each component.
```

This makes it crystal clear that:
- The array pointer comes from `rust_hmalloc` (libc allocator) → must use `rust_hfree`
- The string pointers come from `CString::into_raw` (Rust allocator) → must use `CString::from_raw`

### 2. Removed dead null-check branch (lines 136-144 removed)

Deleted the unreachable error-handling block:

```rust
// REMOVED:
```

## Notes

This phase addresses the subtle but critical issue of allocator-family matching: memory allocated by one allocator (e.g., Rust's global allocator) must be freed by the same allocator, not a different one (e.g., libc's malloc/free). Mixing them causes undefined behavior.

The `copy_argv_to_c` function is particularly tricky because it allocates memory from *two different allocator families*:
1. Array of pointers: `rust_hmalloc` (libc) → must free with `rust_hfree` (libc)
2. String contents: `CString::into_raw` (Rust) → must reclaim with `CString::from_raw` (Rust)

The updated documentation now makes this explicit and prevents future confusion.
