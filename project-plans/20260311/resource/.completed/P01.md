**Plan**: PLAN-20260314-RESOURCE.P01

## Scope

This phase revalidated all 11 planned gaps against the actual authoritative Rust resource implementation:

- `rust/src/resource/ffi_bridge.rs`
- `rust/src/resource/dispatch.rs`
- `rust/src/resource/type_registry.rs`

## Executive Summary

**Final verdict: REVISED**

The plan is directionally correct, but it needs revision before implementation proceeds unchanged.

### Summary outcome

- **CONFIRMED as real gaps:** GAP-1 through GAP-11
- **But several need correction/expansion:** GAP-1, GAP-3, GAP-4, GAP-5, GAP-6, GAP-7, GAP-9, GAP-11
- **Additional gaps discovered:** 5 material additions

### High-level findings

1. The planned 11 gaps are all based on real code issues or real cleanup work.
2. However, some gap descriptions understate the real root cause or miss adjacent affected behavior.
3. The biggest missed issue is that **value-type handling is broken more broadly than the plan states**, especially for `STRING` and `UNKNOWNRES` under `res_GetResource()` semantics.
4. The plan also misses a **mismatch between `res_type` and `type_handler_key` during save filtering**, which materially affects GAP-7.
5. The dead-code cleanup target list is incomplete because `mod.rs` still publicly exports those non-authoritative modules.

## GAP-1: `res_OpenResFile` missing directory sentinel detection

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-FILE-002, REQ-RES-FILE-003  
**Files:** `rust/src/resource/ffi_bridge.rs`

### Evidence

`ffi_bridge.rs:28-38` extern block declares UIO functions, but **does not declare `uio_stat`**:

- `uio_fopen` at line 30
- `uio_fclose` at line 31
- `uio_fread` at line 32
- `uio_fwrite` at line 33
- `uio_fseek` at line 34
- `uio_ftell` at line 35
- `uio_fgetc` at line 36
- `uio_fputc` at line 37
- `uio_unlink` at line 38
- **no `uio_stat` present**

`ffi_bridge.rs:994-1002`:

```rust
pub unsafe extern "C" fn res_OpenResFile(
    dir: *mut c_void,
    filename: *const c_char,
    mode: *const c_char,

    if filename.is_null() || mode.is_null() {
        return ptr::null_mut();
    }
    uio_fopen(dir, filename, mode)
```

This directly calls `uio_fopen()` and never checks directory status.

### Analysis

The gap is real exactly as described: no `uio_stat`, no directory detection, no `STREAM_SENTINEL` return.

### Revision needed

**Expand GAP-1 scope:** the plan should explicitly record that this is not only a missing extern or missing branch in `res_OpenResFile`; it is part of a larger sentinel-policy gap also affecting `LoadResourceFromPath` (see GAP-9 and Additional Gap A1).

## GAP-3: `UNKNOWNRES` stored as heap type instead of value type

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-UNK-001, REQ-RES-UNK-003, REQ-RES-LOAD-011  
**Files:** `rust/src/resource/dispatch.rs`, `rust/src/resource/ffi_bridge.rs`, `rust/src/resource/type_registry.rs`

### Evidence

`ffi_bridge.rs:199-201`:

```rust
dispatch
    .type_registry
    .install("UNKNOWNRES", None, None, None);
```

So `UNKNOWNRES` has:
- no `load_fun`
- no `free_fun`
- no `to_string`

`dispatch.rs:80-89`:

```rust
let (handler_key, is_value_type) = if let Some(h) = handlers {
    let is_value = h.free_fun.is_none();
    (type_name.to_string(), is_value)
} else {
    log::warn!(...);
    ("UNKNOWNRES".to_string(), false)
};
```

For unknown type fallback, `is_value_type` is explicitly set to `false`.

`dispatch.rs:95-105` only eagerly loads value types:

```rust
if is_value_type {
    if let Some(h) = self.type_registry.lookup(type_name) {
        if let Some(load_fn) = h.load_fun {
            ...
        }
    }
}
```

But in the unknown-type case, `type_registry.lookup(type_name)` is already `None`, so even if `is_value_type` were corrected, the eager-load path still uses the wrong lookup key.

### Analysis

The planned gap is real, but **the plan understates the root cause**.

There are actually **three linked defects**:

1. `UNKNOWNRES` is registered without a load function.
2. Unknown fallback marks `is_value_type = false`.
3. The eager-load path looks up `type_name`, not `handler_key`, so an unknown type would still not load even if `handler_key = "UNKNOWNRES"` and `is_value_type = true` were set.

### Revision needed

**Revise GAP-3** to include the third issue above. The minimal fix is not just:
- register `UNKNOWNRES` with a load function, and
- mark it value type,

but also:
- use `handler_key` for the eager value-type lookup path in `process_resource_desc`.

## GAP-4: `get_resource` doesn't handle value types correctly

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-LOAD-011, REQ-RES-LOAD-003  
**Files:** `rust/src/resource/dispatch.rs`

### Evidence

`dispatch.rs:137-167`:

```rust
let ptr_is_null = unsafe { desc.data.ptr.is_null() };
if ptr_is_null {
    let load_fn = self
        .type_registry
        .lookup(&desc.type_handler_key)
        .and_then(|h| h.load_fun);
    ...
}

let desc = self.entries.get_mut(key).unwrap();
let ptr = unsafe { desc.data.ptr };
if ptr.is_null() {
    return ptr::null_mut();
}

desc.refcount += 1;
ptr
```

This logic is heap-pointer centric:
- it interprets `data.ptr.is_null()` as “not loaded yet”
- it interprets `data.ptr` as the only success path
- it returns null whenever `data.ptr` is null

For value types, `data` is stored in either:
- `data.num`, or
- `data.str_ptr`

So this method cannot correctly return value-type resources.

### Analysis

The gap is real.

It also affects more than the plan text emphasizes:

- `STRING` value types are broken here.
- `INT32`, `BOOLEAN`, `COLOR` are broken here.
- `UNKNOWNRES` general accessor behavior is broken here.

### Revision needed

**Expand GAP-4 description** to explicitly state that `STRING` and `UNKNOWNRES` should return `str_ptr`, and numeric value types should return `num as *mut c_void`, with `refcount += 1` in all success cases.

## GAP-5: `UninitResourceSystem` doesn't free loaded heap resources

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-LIFE-004, REQ-RES-OWN-005, REQ-RES-OWN-010  
**Files:** `rust/src/resource/ffi_bridge.rs`, `rust/src/resource/dispatch.rs`

### Evidence

`ffi_bridge.rs:293-303`:

```rust
pub extern "C" fn UninitResourceSystem() {
    let mut guard = match RESOURCE_STATE.lock() {
        Ok(g) => g,
        Err(poisoned) => poisoned.into_inner(),

    if guard.is_none() {
        log::warn!("UninitResourceSystem called when not initialized");
```

There is no iteration over entries and no `free_fun` invocation.

`dispatch.rs:44` confirms entries are available via `pub entries: HashMap<String, FullResourceDesc>`.

`dispatch.rs:188-191`, `295-298` show the free-function lookup pattern the system already uses elsewhere.

### Analysis

The gap is fully real.

### Revision needed

**Expand GAP-5** to include warning behavior for live refcounts, because replacement/removal semantics already log warnings for live resources and shutdown cleanup should likely preserve diagnostic context as well. This is not a reason to block the gap, but it is part of a complete parity fix.

## GAP-6: Entry replacement doesn't free old loaded heap resources

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-OWN-009, REQ-RES-IDX-006  
**Files:** `rust/src/resource/dispatch.rs`

### Evidence

`dispatch.rs:108-117`:

```rust
let desc = FullResourceDesc {
    ...
};

self.entries.insert(key.to_string(), desc);
```

No old entry is examined before replacement.

By contrast, `dispatch.rs:283-307` in `remove_resource()` explicitly looks up current entry state and conditionally calls `free_fun` before removal.

### Analysis

The leak is real for any replaced loaded heap resource.

### Revision needed

**Expand GAP-6**: in addition to freeing the old loaded heap resource, replacement must consider cached C-string state in `ffi_bridge.rs` (`string_cache`, `type_cache`) because those caches are keyed by resource name and can persist stale values after type/value changes. That issue is not directly in `dispatch.rs`, but replacement semantics interact with it. See Additional Gap A4.

## GAP-7: `SaveResourceIndex` emits entries without `toString`

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-IDX-005, REQ-RES-UNK-002  
**Files:** `rust/src/resource/ffi_bridge.rs`

### Evidence

`ffi_bridge.rs:423-439`:

```rust
let serialized = if desc.res_type == "STRING" {
    format!("{}:{}", desc.res_type, desc.fname)
} else if let Some(handlers) = state.dispatch.type_registry.lookup(&desc.res_type) {
    if let Some(to_string_fn) = handlers.to_string {
        ...
        format!("{}:{}", desc.res_type, value_str)
    } else {
        format!("{}:{}", desc.res_type, desc.fname)
    }
} else {
    format!("{}:{}", desc.res_type, desc.fname)
};
```

This clearly serializes entries even when:
- handler exists but `to_string` is `None`, or
- no handler exists for `desc.res_type`

### Analysis

The gap is real.

But there is also a subtle adjacent issue: for unknown types, entries preserve the original `res_type` string while `type_handler_key` is `UNKNOWNRES`. Save logic looks up by `desc.res_type`, not `type_handler_key`, so unknown fallback entries will be treated as “no handler found” and still serialized via fallback. That behavior is wrong under REQ-RES-UNK-002 and is part of the same bug family.

### Revision needed

**Revise GAP-7** to explicitly mention the `res_type` vs `type_handler_key` mismatch. The save-eligibility check should be based on the authoritative active handler identity, not only the preserved original type name.

## GAP-9: `LoadResourceFromPath` sentinel/zero-length guards incomplete

**Plan status:** CONFIRMED  
**Relevant requirements:** REQ-RES-FILE-005, REQ-RES-FILE-008  
**Files:** `rust/src/resource/ffi_bridge.rs`

### Evidence

`ffi_bridge.rs:1132-1165`:

```rust
pub unsafe extern "C" fn LoadResourceFromPath(
    pathname: *const c_char,
    load_fn: Option<ResourceLoadFileFun>,
) -> *mut c_void {
    if pathname.is_null() {
        return ptr::null_mut();
    }
    let load_fn = match load_fn {
        Some(f) => f,
        None => return ptr::null_mut(),
    };
    let mode = b"rb\0".as_ptr() as *const c_char;
    let fp = uio_fopen(contentDir, pathname, mode);
    if fp.is_null() {
        return ptr::null_mut();
    }
    uio_fseek(fp, 0, 2);
    let length = uio_ftell(fp) as u32;
    uio_fseek(fp, 0, 0);
    _cur_resfile_name = pathname;
    let result = load_fn(fp, length);
    _cur_resfile_name = ptr::null();
    uio_fclose(fp);
    result
}
```

Problems visible here:

1. Opens with `uio_fopen` directly, not `res_OpenResFile`.
2. No sentinel detection path at all.
3. No `length == 0` guard.
4. No guard for early exits after open except null-open case.

### Analysis

The gap is real.

The plan understates it: this is not just “zero-length guard incomplete.” It is also structurally bypassing the resource open helper and therefore bypassing directory-sentinel policy entirely.

### Revision needed

**Revise GAP-9** to explicitly include:
- using `res_OpenResFile` or equivalent sentinel-aware open logic,
- rejecting `STREAM_SENTINEL` before callback invocation,
- rejecting zero-length files before callback invocation.

## A2: `LoadResourceFromPath` does not restore `_cur_resfile_name` on all failure paths after open

**Status:** ADDITIONAL GAP  
**Files:** `rust/src/resource/ffi_bridge.rs`  
**Related planned gaps:** GAP-9

### Evidence

`ffi_bridge.rs:1156-1164` sets and clears `_cur_resfile_name` only around the happy callback path:

```rust
_cur_resfile_name = pathname;
```

There is no scoped guard or RAII cleanup structure protecting this state if future edits insert early returns/panics after the guard is set. The current code path is linear, but the requirement is explicit about all exit paths.

### Why this matters

This is not a separate user-visible bug today in the exact current branch structure, but it is a structural conformance gap relative to REQ-RES-FILE-004 because the helper does not actually encode “restore on every exit path” as an invariant.

- REQ-RES-FILE-004
