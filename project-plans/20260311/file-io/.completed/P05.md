### TASK 3: errno Setting [OK]
**Requirement**: REQ-FIO-ERRNO

**Enhanced errno support**:
- `set_errno()` helper already existed from P04 (line 37) using `libc::__error()` for macOS
- Added `fail_errno<T>()` helper for convenient error returns with errno setting
- Updated all file operation functions to set appropriate errno values:
  - **uio_rename**: Sets EINVAL for null arguments, preserves OS errno on fs::rename failure
  - **uio_access**: Sets EINVAL for null dir, ENOENT for missing files
  - **uio_stat**: Sets EINVAL for null arguments, preserves OS errno on metadata failure (typically ENOENT)
  - **uio_mkdir**: Sets EINVAL for null dir, preserves OS errno on create_dir failure (including EEXIST)
  - **uio_rmdir**: Sets EINVAL for null dir, preserves OS errno on remove_dir failure
  - **uio_unlink**: Sets EINVAL for null dir, preserves OS errno on remove_file failure
  - **uio_fopen**: Sets EINVAL for null arguments and invalid mode strings, preserves OS errno on file open failure
  - **uio_open**: Sets EINVAL for null arguments, preserves OS errno on file open failure (including ENOENT)
  - **uio_getFileLocation**: Sets ENOENT for missing files, ENOMEM for allocation failures

**Error code mapping strategy**:
- Use `e.raw_os_error().unwrap_or(libc::EIO)` to preserve platform-specific errno when available
- Fall back to `EIO` for unexpected errors without OS error codes
- Use specific codes (EINVAL, ENOENT, EEXIST, ENOMEM) for well-defined failure cases

**Tests Added**:
- `test_uio_access_sets_enoent_on_missing_file`: Validates ENOENT on nonexistent file
- `test_uio_mkdir_sets_eexist_on_existing_dir`: Validates EEXIST when directory already exists
- `test_uio_fopen_sets_einval_on_invalid_mode`: Validates EINVAL for invalid mode string ("x")
- `test_uio_open_sets_enoent_on_missing_file`: Validates ENOENT when opening nonexistent file

### TASK 4: Panic Safety [OK]
**Requirement**: REQ-FIO-PANIC-SAFETY

Implemented panic containment strategy:
- Added `ffi_guard!` macro using `std::panic::catch_unwind` with `AssertUnwindSafe`
- Macro takes default return value and function body
- On panic, sets errno to EIO and returns the safe default value
- Applied to all exported `extern "C"` functions updated in this phase:
  - `uio_rename`: default `-1`
  - `uio_access`: default `-1`
  - `uio_stat`: default `-1`
  - `uio_mkdir`: default `-1`
  - `uio_rmdir`: default `-1`
  - `uio_unlink`: default `-1`
  - `uio_fopen`: default `ptr::null_mut()`
  - `uio_open`: default `ptr::null_mut()`
  - `uio_getFileLocation`: default `-1`

**Note**: The `uio_fgets` function already had panic containment from P04a (lines 622-634), demonstrating the pattern was already established.

## errno/Validation Coverage Notes

All major file operations now properly set errno:
- **EINVAL**: Invalid arguments (null pointers, invalid mode strings, bad parameters)
- **ENOENT**: File/directory not found
- **EEXIST**: Directory already exists (mkdir)
- **ENOMEM**: Memory allocation failure
- **EIO**: General I/O errors or panic-contained failures
- **Preserved OS codes**: fs::rename, fs::metadata, fs::create_dir, fs::remove_dir, fs::remove_file, OpenOptions::open all preserve platform-specific error codes

## Panic-Containment Strategy Summary

**Pattern Established**: `ffi_guard!` macro
- Wraps function body in `catch_unwind`
- Uses `AssertUnwindSafe` for proper panic boundary
- Sets errno to EIO on panic
- Returns safe default value based on function signature:
  - Pointers: `ptr::null_mut()`
  - Integers: `-1`
  - (Future: EOF for stream ops, 0 for size_t as needed)

**Coverage**: All updated functions in P05 are panic-safe. Previous P04a work on `uio_fgets` already demonstrated this pattern.

**Philosophy**: No panic may cross the FFI boundary. All panics are caught and converted to safe, diagnosable error returns.
