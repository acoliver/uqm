## G1: Thread Return Value Propagation

### Result
PASS WITH CORRECTIONS

### Verified actual code path
Actual current data flow in source is:

1. `CreateThread_Core()` in `sc2/src/libs/threads/rust_thrcommon.c` allocates `TrueThread` and `ThreadStartInfo`, then calls:
   - `thread->native = rust_thread_spawn(name, RustThreadHelper, startInfo);`
2. `rust_thread_spawn()` in `rust/src/threading/mod.rs` calls `spawn_c_thread(...)`.
3. `spawn_c_thread(...)` currently returns `Result<Thread<()>>` and runs this closure:
   - `let _ = unsafe { func(data) };`
   This discards the C thread function result.
4. The C entry point is `RustThreadHelper()` in `rust_thrcommon.c`, which does:
   - `result = (*func)(data);`
   - `FinishThread((Thread) thread);`
   - `return result;`
   So the C shim does capture and return the entry function's integer status.
5. `rust_thread_join(thread)` in `mod.rs` converts the opaque pointer to `Box<Thread<()>>`, joins it, and returns:
   - `1` on successful join
   - `0` on failure
6. `WaitThread()` in `rust_thrcommon.c` does:
   - `int result = rust_thread_join(t->native);`
   - `*status = result;`
   Therefore it writes join success/failure, not thread exit status.

The plan's high-level claim is correct: return value propagation is broken at the Rust FFI adapter boundary.

Confirmed real current code matches the old-code target list in substance:

That is directionally right, but the variable naming in the plan is misleading because the current local `result` is a join-success flag. The corrected post-change structure should distinguish:

## G3: StartThread_Core Detached Reference-Design Mismatch

### Result
PASS WITH CORRECTION

### Verified actual code
Current `StartThread_Core()` in `sc2/src/libs/threads/rust_thrcommon.c`:
- allocates `TrueThread` via `AllocThreadHandle(name)`
- allocates `ThreadStartInfo`
- stores `startInfo->thread = thread`
- calls `thread->native = rust_thread_spawn(name, RustThreadHelper, startInfo);`
- if spawn fails, frees `startInfo` and `thread`
- otherwise returns without exposing the thread handle to caller

That matches the plan's claim: it does not use `rust_thread_spawn_detached()`, and the lifecycle path depends on retaining the handle internally.

The verified dependency chain is also accurate:
1. `RustThreadHelper()` calls `FinishThread((Thread) thread)`.
2. `FinishThread()` enqueues the `TrueThread` in `pendingDeath[]`.
3. `ProcessThreadLifecycles()` iterates `pendingDeath[]` and calls `WaitThread(t, NULL)`.
4. `WaitThread()` only joins if `t && t->native`.

So switching `StartThread_Core()` to `rust_thread_spawn_detached()` without redesign would break lifecycle cleanup because there would be no joinable `RustThread*` stored in `thread->native`.

### Verified old code to replace
There is no code replacement needed here if the conclusion remains "keep current spawn path"; this is mainly a documentation/spec-alignment gap.

### Corrections needed to the plan
One important correction: the plan says the caller does not receive a joinable handle, which is true, but the code still creates a hidden joinable handle for lifecycle cleanup. The real mismatch is not merely documentation style; it is a structural contradiction between detached API intent and actual cleanup mechanism.

### Additional integration callers found
Actual `StartThread(...)` search found one live call site:

This caller was not listed in the plan's integration touchpoints and should be added.

That means:

## G7: Stale Comment in C Adapter

### Result
PASS

### Verified actual code
The comment in `sc2/src/libs/threads/rust_thrcommon.c` is present exactly as described:

`/* Rust std::sync::Mutex is not recursive; using regular mutex */`

It sits directly above `CreateRecursiveMutex_Core()`.

That comment is inaccurate with respect to actual current implementation because `CreateRecursiveMutex_Core()` returns `rust_mutex_create(name)`, and the Rust backing type for that FFI mutex is `RustFfiMutex`, which implements recursive behavior through:
- owner thread tracking
- depth counting
- `lock()`, `try_lock()`, `unlock()`, and `depth()` methods

This is confirmed in `rust/src/threading/mod.rs` under `RustFfiMutex`.

### Verified old code to replace
Confirmed comment location and contradiction with implementation.

### Corrections needed to the plan
One nuance to preserve: the same `rust_mutex_create()` implementation backs both plain mutexes and recursive mutexes in the Rust adapter. So although the comment above `CreateRecursiveMutex_Core()` is wrong, the broader plain-mutex semantics are also materially different from classic non-recursive mutex behavior in the current Rust implementation. The plan is right to avoid declaring the plain-mutex contract resolved.

### Evidence
- `sc2/src/libs/threads/rust_thrcommon.c`: `CreateRecursiveMutex_Core()` comment
- `rust/src/threading/mod.rs`: `RustFfiMutex` implementation and `rust_mutex_create()`
