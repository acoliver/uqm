# Phase 09.75: Build/Feature Coupling Enforcement

## Phase ID
`PLAN-20260314-AUDIO-HEART.P09.75`

## Prerequisites
- Required: Phase P09.5 completed
- Expected: Comm handshake integration verified; authoritative build entry points ready to inspect/update

## Purpose

Own the normative build-coupling contract from the specification and requirements so the Rust audio-heart replacement cannot be enabled in a silently inconsistent way. Preflight only checks whether the coupling appears to exist; this phase must implement or harden the coupling itself and prove the resulting build behavior.

## Requirements Implemented (Expanded)

### Build coupling contract
**Requirement text**: The C preprocessor macro `USE_RUST_AUDIO_HEART` and the Cargo feature `audio_heart` must be enabled together.

Behavior contract:
- GIVEN: the authoritative build path enables `USE_RUST_AUDIO_HEART`
- WHEN: the Rust crate/build is invoked
- THEN: Cargo is passed `--features audio_heart` (or an equivalent authoritative feature-selection mechanism)

- GIVEN: the authoritative build path enables Cargo `audio_heart`
- WHEN: the C side is compiled/linked
- THEN: `USE_RUST_AUDIO_HEART` is also enabled for the matching build

### Fast-fail mismatch handling
**Requirement text**: If the build enables the replacement declarations without enabling the corresponding exported implementation symbols, then the build or integration configuration shall fail rather than producing a silently inconsistent binary.

Behavior contract:
- GIVEN: `USE_RUST_AUDIO_HEART` is enabled without Cargo `audio_heart`
- WHEN: the authoritative build command runs
- THEN: the build fails fast with a deterministic, reviewable signal

- GIVEN: Cargo `audio_heart` is enabled without the matching C-side replacement path
- WHEN: the authoritative build command runs
- THEN: the build fails fast or rejects the unsupported configuration explicitly

Why it matters:
- This is an ABI/build-integration contract, not optional hygiene
- Preflight discovery is insufficient if the build graph can still drift into an inconsistent state later
- Functional phases cannot be trusted if the replacement path can compile half-enabled

### Files to modify
Modify only the actual authoritative build/config files identified above. Candidate areas include, but are not limited to:
- root build scripts / Makefiles / cargo wrapper scripts
- `sc2/config_unix.h` or the config source that owns `USE_RUST_AUDIO_HEART`
- Rust/C build glue that forwards Cargo features
- plan documentation in this file and the tracker/overview only if needed to reflect the enforced contract
